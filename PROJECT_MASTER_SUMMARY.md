# Home-Energetika

## Назначение проекта

**Home-Energetika** — единая инженерная платформа для развития интернет-магазина насосного оборудования.

Проект состоит из независимых подсистем, каждая из которых решает собственную задачу, имеет самостоятельную архитектуру, документацию и жизненный цикл.

Все подсистемы работают с единым каталогом OpenCart и взаимодействуют через четко определенные контракты.

---

# Архитектура проекта

На текущий момент проект состоит из нескольких основных подсистем.

## Pump Selector

Назначение:

Разработка интеллектуальных подборщиков оборудования.

Подсистема отвечает за:

* алгоритмы подбора;
* пользовательский интерфейс;
* бизнес-логику;
* интеграцию с OpenCart.

Подборщик использует стандартизированные данные каталога и не занимается их исправлением.

---

## Catalog Standardization

Назначение:

Управление качеством каталога.

Подсистема отвечает за:

* стандартизацию характеристик;
* унификацию атрибутов;
* импорт данных производителей;
* автоматизацию обработки каталогов;
* поддержку единого стандарта хранения данных.

Результатом работы подсистемы является каталог, пригодный для использования всеми остальными подсистемами проекта.

---

## Framework Standardization

Назначение:

Безопасная стандартизация характеристик как controlled attribute consolidation workflow.

Подсистема отвечает за:

* DB-readonly discovery характеристик;
* подготовку списка кандидатов по реальным `attribute_id` и `attribute_name`;
* human canonical selection;
* explicit include/exclude alias decision;
* raw values inventory;
* canonical unit / `normalized_value` contract;
* normalization proposals;
* standalone review-chain before any apply-plan.

Framework Standardization не является fully automatic normalizer.

Похожие `attribute_name` не объединяются автоматически. Canonical attribute selection делает пользователь.

`config/jobs` не является стартовой точкой угадывания характеристики. `config/jobs` должен быть результатом accepted canonical decision/contract.

`approved` в review-chain не означает SQL apply permission.

Новый ChatGPT-чат или новый участник проекта должен начинать с:

```text
framework-standardization/docs/START_HERE.md
```

# Взаимодействие подсистем

```text
                 Производители
                       │
                       ▼
         Catalog Standardization
                       │
                       ▼
        Стандартизированный каталог
                       │
                       ▼
              Pump Selector
                       │
                       ▼
                 Пользователь
```

Каждая подсистема имеет единственную ответственность.

Подсистемы взаимодействуют через стандартизированные данные, а не через внутреннюю реализацию друг друга.

---

# Архитектурные принципы

## Разделение ответственности

Каждая подсистема отвечает только за собственную предметную область.

Изменение логики одной подсистемы не должно требовать изменения архитектуры другой.

---

## Документация является частью архитектуры

Документация разрабатывается одновременно с проектом.

Архитектурные решения фиксируются до начала реализации.

---

## Стандарты предшествуют реализации

Сначала принимается стандарт.

После этого изменяются данные, код или алгоритмы.

---

## Эволюционное развитие

Архитектура развивается на основе практического опыта.

Новые подсистемы выделяются только тогда, когда появляется самостоятельная предметная область, собственный жизненный цикл и независимые архитектурные решения.

---

# Архитектура документации

Документация проекта имеет иерархическую структуру.

```
README.md
        │
        ▼
PROJECT_MASTER_SUMMARY.md
        │
        ▼
PROJECT_MASTER_SUMMARY.md выбранной подсистемы
        │
        ▼
Специализированные документы подсистемы
```

Каждая подсистема самостоятельно определяет набор специализированных документов, необходимых для своей работы.

Для активных подсистем дополнительно могут использоваться:

* `docs/START_HERE.md` — быстрый onboarding entrypoint;
* `docs/HANDOFF.md` — оперативное состояние активной разработки;
* `docs/DECISIONS.md` — зафиксированные архитектурные решения;
* `docs/RUNTIME_CHECKS.md` — история ручных проверок и runtime checks;
* `docs/RULES.md` — правила процесса, если подсистема их использует;
* specialized specs — документы по отдельным boundaries, components и workflow steps.

---

# Правила создания новых подсистем

Новая подсистема создается только при наличии:

* самостоятельной предметной области;
* собственной архитектуры;
* собственного жизненного цикла;
* независимой документации.

При создании новой подсистемы обязательны:

• README.md
• PROJECT_MASTER_SUMMARY.md

Дополнительно создаются по мере необходимости:

• HANDOFF.md (только при активной разработке)
• специализированные документы (CATALOG_STANDARD, DATA_PIPELINE и т.д.)

---

# Масштабируемость проекта

Архитектура проекта допускает независимое развитие подсистем.

На текущем этапе основными направлениями являются:

* развитие алгоритмов подбора оборудования;
* стандартизация каталога;
* автоматизация импорта данных производителей.

По мере развития проекта могут появляться новые подсистемы без изменения существующей архитектуры.
