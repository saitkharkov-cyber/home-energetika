# Pump Selector

## Назначение

**Pump Selector** — подсистема проекта **Home-Energetika**, предназначенная для интеллектуального подбора насосного оборудования по заданным параметрам пользователя.

Подсистема использует стандартизированные данные каталога OpenCart и формирует рекомендации по выбору оборудования на основе бизнес-логики подбора.

Pump Selector не выполняет нормализацию данных каталога. Он работает исключительно с подготовленными данными, предоставляемыми подсистемой Catalog Standardization.

# Основные задачи

Подсистема отвечает за:

- подбор оборудования по заданным параметрам;
- реализацию алгоритмов подбора;
- формирование рекомендаций Best Price, Optimal и Premium;
- интеграцию с OpenCart;
- отображение результатов пользователю.

---

# Основные архитектурные принципы

Подсистема построена на следующих принципах:

- Source of Truth для каталожных данных — OpenCart.
- Нормализация выполняется только Catalog Standardization.
- Builder строит технический cache.
- Selector работает исключительно с готовым cache.
- Builder не принимает решений о рекомендациях.
- Selector не исправляет данные каталога.
- Архитектурные контракты документируются отдельно от реализации.
- Бизнес-логика отделена от технической реализации.

---

# Архитектурный принцип

Pump Selector не зависит от конкретного производителя оборудования.

Алгоритмы работают с унифицированными характеристиками, предоставляемыми Catalog Standardization.

Это позволяет:

- добавлять новых производителей;
- расширять поддерживаемые категории оборудования;
- совершенствовать алгоритмы подбора без изменения архитектуры системы.

---

# Статус проекта

Подсистема находится в активной разработке и используется в составе проекта Home-Energetika.

Архитектура ориентирована на постепенное расширение функциональности при сохранении обратной совместимости существующих компонентов и контрактов.

# Правило документации

Документация отражает текущее состояние проекта.

При изменении архитектуры соответствующие документы должны обновляться одновременно с изменением кода.

История принятых решений, исследований и устаревших материалов хранится отдельно в каталоге:

```text

```

Актуальная документация всегда описывает текущее состояние системы.
