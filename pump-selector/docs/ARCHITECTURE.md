# Architecture

## Назначение

Документ описывает текущую архитектуру проекта `pump-selector`: основные компоненты, зоны ответственности, поток данных и границы между нормализацией, построением cache и пользовательским подбором.

Детальная бизнес-логика выбора `Best Price`, `Optimal` и `Premium` описывается отдельно в `docs/SELECTION_STRATEGY.md`.

## Общая идея

`pump-selector` — модуль подбора скважинных насосов для OpenCart.

Пользователь вводит параметры системы, модуль рассчитывает требуемые характеристики насоса и выбирает подходящие товары из каталога.

Чтобы не выполнять тяжёлую обработку сырых данных при каждом пользовательском запросе, используется служебная cache-таблица `pump_selector_product`.

## Поток данных

```text
OpenCart Catalog
  -> Catalog Standardization
  -> PumpSelectorCacheBuilder
  -> pump_selector_product
  -> PumpSelector
  -> prepareProductCardData()
  -> OpenCart Catalog
  -> UI
```

## Компоненты

### OpenCart Catalog

Стандартный каталог OpenCart является источником истины для товарных данных: названия, производителя, изображения, цены, наличия, статуса, ссылок и карточек товара.

Подборщик не должен заменять каталог OpenCart собственной копией товарных данных.

### Catalog Standardization

`Catalog Standardization` отвечает за подготовку характеристик каталога к использованию подборщиком.

Ответственность:

- нормализация значений;
- унификация единиц измерения;
- использование канонических `attribute_id`;
- подготовка данных для Cache Builder.

Cache Builder не выполняет повторную нормализацию строк.

### PumpSelectorCacheBuilder

`PumpSelectorCacheBuilder` — служебный backend-компонент, который строит таблицу `pump_selector_product`.

Он выполняет:

1. очистку текущего cache;
2. загрузку товаров из целевых категорий;
3. чтение атрибутов по `attribute_id`;
4. проверку базовой пригодности товара;
5. приведение подготовленных значений к нужным типам;
6. вычисление `brand_priority`;
7. запись готовой строки в cache.

Builder не участвует в пользовательском подборе напрямую.

### pump_selector_product

`pump_selector_product` — служебная cache-таблица подборщика.

Она не является копией каталога OpenCart.

В cache хранятся только данные, необходимые для отбора и ранжирования:

- `product_id`;
- `max_head_m`;
- `max_flow_l_min`;
- `pump_diameter_mm`;
- `voltage`;
- `brand_priority`;
- `is_eligible`;
- `product_price`;
- `quantity`;
- `status`.

Display-данные товара (`name`, `manufacturer`, `image`) не должны использоваться из cache. Они подтягиваются из OpenCart после выбора `product_id`.

### PumpSelector

`PumpSelector` отвечает за пользовательский подбор:

1. принимает пользовательский ввод;
2. рассчитывает требования;
3. ищет кандидатов в `pump_selector_product`;
4. выбирает роли рекомендаций;
5. получает актуальные данные карточек из OpenCart;
6. передаёт результат в шаблон.

### Product Card Enrichment

После выбора `product_id` selector получает из OpenCart:

- название;
- производителя;
- изображение;
- актуальную цену;
- наличие;
- ссылку;
- прочие UI-данные.

## Границы ответственности

### Catalog Standardization отвечает за

- нормализацию характеристик;
- подготовку исходных данных;
- соответствие характеристик каноническим `attribute_id`.

### Cache Builder отвечает за

- чтение подготовленных данных;
- проверку обязательных атрибутов;
- фильтрацию по базовым условиям;
- вычисление `brand_priority`;
- запись cache.

### PumpSelector отвечает за

- расчёт требований пользователя;
- выбор кандидатов из cache;
- формирование ролей рекомендаций;
- передачу выбранных товаров в слой отображения.

### OpenCart Catalog отвечает за

- актуальные товарные данные;
- отображаемую цену;
- название;
- производителя;
- изображение;
- наличие;
- статус;
- карточку товара.

## Attribute Contract

Подборщик использует канонические `attribute_id`. Поиск характеристик по названиям запрещён.

| Attribute ID | Значение | Cache field |
|---:|---|---|
| `12` | Максимальный напор | `max_head_m` |
| `13` | Максимальная производительность | `max_flow_l_min` |
| `15` | Напряжение | `voltage` |
| `44` | Диаметр насоса | `pump_diameter_mm` |

Обязательные атрибуты: `12`, `13`, `44`.

Атрибут `15` необязательный, но используется при наличии.

## Brand Priority

`brand_priority` используется для ранжирования премиальных кандидатов.

| Производитель | Brand Priority |
|---|---:|
| `Pedrollo*` | `10` |
| `Sumoto*` | `8` |
| `Belamos*` | `5` |
| Остальные | `0` |

`manufacturer` может читаться из OpenCart при построении cache только для вычисления `brand_priority`.

## Recommendation Roles

Подборщик формирует до трёх рекомендаций:

- `Best Price`;
- `Optimal`;
- `Premium`.

Бизнес-смысл и стратегия выбора описаны в `docs/SELECTION_STRATEGY.md`.

## Protected Rebuild Endpoint

Cache Builder должен быть пригоден для вызова из защищённого backend-endpoint.

Endpoint предназначен для ручной или cron-пересборки cache и возвращает JSON-статистику.

## Debug / Trace

Временная диагностика может использоваться для анализа поведения алгоритма выбора.

Диагностика должна:

- не менять бизнес-логику;
- не менять результат подбора;
- работать только на чтение;
- быть легко удаляемой;
- не оставаться в production-коде без необходимости.

## Инварианты архитектуры

- `Catalog Standardization` является источником нормализованных характеристик.
- Cache Builder не выполняет повторную нормализацию строк.
- Cache Builder использует `attribute_id`, а не названия характеристик.
- Пользовательский подбор работает по готовому cache.
- Cache не является копией каталога OpenCart.
- Display-данные товара берутся из OpenCart после выбора `product_id`.
- `pump_selector_product` хранит только данные, необходимые для отбора и ранжирования.
- Бизнес-логика выбора рекомендаций описывается отдельно от технической реализации.
- Временная диагностика не должна оставаться частью production-логики.
- Документация должна разделять текущее состояние и историю решений.

## Что может изменяться

Может изменяться:

- SQL-запросы;
- scoring;
- fallback-ветки;
- структура debug trace;
- cron endpoint;
- способ хранения секрета;
- точные коэффициенты ранжирования;
- правила `brand_priority`.

Не должны нарушаться:

- границы ответственности компонентов;
- источник истины каталога;
- роль cache как service-layer;
- контракт входных и выходных данных;
- бизнес-смысл рекомендаций.
