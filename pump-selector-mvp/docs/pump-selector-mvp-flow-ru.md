# MVP подбора скважинного насоса для OpenCart 2.3

Документ описывает MVP-селектор только для погружных скважинных насосов. Это не профессиональный инженерный калькулятор, а инструмент предварительного подбора. Все рекомендации должны использовать осторожные формулировки: `предварительный подбор`, `обычно подходит`, `требует подтверждения специалистом`.

## 1. Пользовательский сценарий MVP

### Шаг 1. Основные данные по скважине

Пользователь вводит:

- общую глубину скважины, м;
- уровень воды:
  - известен;
  - не знаю;
- если уровень воды известен:
  - динамический уровень воды, м;
  - или ориентировочный уровень воды, м;
- расстояние от скважины до дома, м;
- диаметр обсадной трубы:
  - известен, мм;
  - не знаю;
- напряжение:
  - 220 В;
  - 380 В.

### Шаг 2. Точки водоразбора

Пользователь выбирает:

- самую высокую точку водоразбора:
  - 1 этаж;
  - 2 этаж;
  - 3 этаж;
  - другое значение;
- точки расхода воды:
  - раковина;
  - душ;
  - унитаз;
  - стиральная машина;
  - посудомоечная машина;
  - полив.

### Шаг 3. Расчет

Селектор рассчитывает:

```text
Требуемый напор H = уровень воды + требуемое давление + вертикальный подъем + потери в трубе
Требуемая подача Q = сумма выбранных точек расхода
```

Расчет остается технической частью результата. На главном экране результата покупателю сначала показывается понятная сводка введенных условий, а не значения `H` и `Q`.

### Шаг 4. Подбор товаров

Товары фильтруются по условиям:

- `max_head_m >= H`;
- `max_flow_l_min >= Q`;
- `voltage = выбранное напряжение`;
- `price > 0`;
- `stock > 0`;
- если диаметр обсадной трубы известен: `pump_diameter_mm <= диаметр обсадной трубы`;
- если диаметр обсадной трубы неизвестен: жесткий фильтр по диаметру не применяется, но показывается предупреждение.

### Шаг 5. Экран результатов

Результаты показываются группами:

1. Лучшая цена.
2. Оптимальный выбор.
3. С запасом.
4. Премиум / брендовый вариант, если такие данные доступны.

Все рекомендации должны быть явно обозначены как предварительный подбор.

Пример формулировки:

> Этот насос обычно подходит для введенных условий. Это предварительный подбор; окончательный выбор требует подтверждения специалистом.

## 2. Правила расчета

### Требуемый напор

```text
H = уровень_воды + требуемое_давление + вертикальный_подъем + потери_в_трубе
```

### Уровень воды

Если известен динамический уровень воды:

```text
уровень_воды = динамический уровень воды
```

Если известен только ориентировочный уровень воды:

```text
уровень_воды = ориентировочный уровень воды
```

Если пользователь выбирает `не знаю`:

```text
уровень_воды = 70% от общей глубины скважины
```

Это оценочный режим при неизвестном уровне воды, а не точный расчет. Его нужно явно показывать пользователю.

Сообщение:

> Уровень воды неизвестен, поэтому селектор использует оценочный режим: рабочий уровень принят как 70% от общей глубины скважины. Результат является предварительным и требует подтверждения специалистом.

### Требуемое давление

Для MVP:

```text
требуемое_давление = 30 м
```

Это приблизительно соответствует 3 бар полезного давления.

Возможная будущая настройка:

- низкое давление: 20 м;
- стандартное давление: 30 м;
- повышенный комфорт: 40 м.

Для MVP по умолчанию используется 30 м.

### Вертикальный подъем

Вертикальный подъем рассчитывается только по полю `самая высокая точка водоразбора`.

| Самая высокая точка водоразбора | Вертикальный подъем |
|---|---:|
| 1 этаж | 3 м |
| 2 этаж | 6 м |
| 3 этаж | 9 м |
| Другое значение | значение пользователя, м |

Если выбрано `другое значение`, пользователь вводит высоту самой высокой точки водоразбора в метрах. Значение должно быть больше 0.

### Потери в трубе

Для MVP используется упрощенная оценка потерь:

```text
потери_в_трубе = расстояние_от_скважины_до_дома * 0.1
```

То есть:

```text
10 м горизонтального расстояния = 1 м потерь напора
```

Также применяется минимальное значение потерь:

```text
потери_в_трубе = max(расстояние * 0.1, 2)
```

Пример:

```text
расстояние = 25 м
потери_в_трубе = 2.5 м
```

### Требуемая подача

Подача рассчитывается как сумма выбранных точек расхода воды.

Рекомендуемые значения для MVP:

| Точка расхода | Подача |
|---|---:|
| Раковина | 8 л/мин |
| Душ | 12 л/мин |
| Унитаз | 6 л/мин |
| Стиральная машина | 10 л/мин |
| Посудомоечная машина | 8 л/мин |
| Полив | 20 л/мин |

Формула:

```text
Q = сумма выбранных точек расхода
```

Пример:

```text
Раковина + душ + унитаз + стиральная машина
Q = 8 + 12 + 6 + 10
Q = 36 л/мин
```

Для MVP не нужно применять сложные коэффициенты одновременности.

Возможное улучшение после MVP:

```text
Если Q > 50 л/мин, можно применить коэффициент одновременности 0.8
```

Но для первой версии проще и безопаснее использовать прямую сумму выбранных точек.

## 3. Правила валидации

### Обязательные поля

Всегда обязательны:

- общая глубина скважины;
- статус уровня воды: известен / не знаю;
- расстояние от скважины до дома;
- самая высокая точка водоразбора;
- минимум одна точка расхода;
- статус диаметра обсадной трубы: известен / не знаю;
- напряжение.

Обязательно только если уровень воды известен:

- динамический или ориентировочный уровень воды.

Обязательно только если диаметр обсадной трубы известен:

- диаметр обсадной трубы, мм.

Обязательно только если выбрано `другое значение` для самой высокой точки водоразбора:

- высота самой высокой точки водоразбора, м.

### Числовая валидация

| Поле | Правило |
|---|---|
| Общая глубина скважины | Больше 0 |
| Уровень воды, если известен | Больше 0 |
| Уровень воды, если известен | Не больше общей глубины скважины |
| Расстояние до дома | 0 или больше |
| Высота самой высокой точки, если указана вручную | Больше 0 |
| Диаметр обсадной трубы, если известен | Больше 0 |

### Предупреждения и блокировки

Если уровень воды неизвестен, расчет разрешен, но нужно показать предупреждение:

> Уровень воды неизвестен, поэтому результат является ориентировочным. Для предварительного подбора используется оценочный режим: 70% от общей глубины скважины. Для более точного подбора желательно измерить динамический уровень воды и подтвердить выбор со специалистом.

Если диаметр обсадной трубы неизвестен, расчет разрешен, но жесткий фильтр по диаметру не применяется. Нужно показать предупреждение:

> Для точного подбора нужно уточнить внутренний диаметр обсадной трубы. Без этого насос может не подойти по габаритам, даже если обычно подходит по напору и подаче.

Если уровень воды больше общей глубины скважины, заблокировать расчет:

> Уровень воды не может быть глубже общей глубины скважины.

Если не выбрана ни одна точка расхода, заблокировать расчет:

> Выберите хотя бы одну точку расхода для расчета требуемой подачи.

Если рассчитанный напор слишком высокий, например больше 120 м, показать предупреждение:

> Требуемый напор получился высоким. Проверьте введенные данные и подтвердите подбор со специалистом.

Если рассчитанная подача слишком высокая, например больше 100 л/мин, показать предупреждение:

> Требуемая подача получилась высокой для бытовой скважинной системы. Проверьте введенные данные и подтвердите подбор со специалистом.

## 4. Логика фильтрации товаров

В MVP используются только товары с нормализованными параметрами:

- `max_head_m`;
- `max_flow_l_min`;
- `pump_diameter_mm`;
- `voltage`;
- `price`;
- `stock`.

Базовое условие фильтрации:

```text
max_head_m >= required_H
max_flow_l_min >= required_Q
voltage = selected_voltage
price > 0
stock > 0
```

Если диаметр обсадной трубы известен, дополнительно применяется:

```text
pump_diameter_mm <= casing_diameter
```

Если диаметр обсадной трубы неизвестен:

- фильтр `pump_diameter_mm <= casing_diameter` не применяется;
- результат помечается предупреждением;
- карточки товаров должны показывать диаметр насоса, чтобы покупатель видел, что параметр нужно уточнить.

Важное уточнение:

Сравнение должно искать насосы с достаточными или большими параметрами, а не точное совпадение.

Правильно:

```text
max_head_m >= H
max_flow_l_min >= Q
```

Неправильно:

```text
max_head_m = H
max_flow_l_min = Q
```

## 5. Логика сортировки

Для каждого подходящего товара рассчитываются запасы:

```text
запас_напора = max_head_m - required_H
запас_подачи = max_flow_l_min - required_Q
```

После фильтрации все запасы должны быть положительными или равными нулю.

Основная сортировка:

```text
1. минимальный запас напора
2. минимальный достаточный запас подачи
3. минимальная цена
```

Так первым будет насос, который обычно подходит под расчетные условия с минимальным достаточным запасом.

Рекомендуемый вариант:

```text
ORDER BY
  (max_head_m - required_H) ASC,
  (max_flow_l_min - required_Q) ASC,
  price ASC
```

Для 90 товаров сортировка в PHP допустима для MVP, но источник данных должен быть нормализованным.

## 6. Структура экрана результатов

### Заголовок

Показать:

```text
Предварительный подбор скважинного насоса
```

Подзаголовок:

```text
На основе введенных данных эти насосы обычно подходят под ваши условия. Это предварительный подбор, а не профессиональный инженерный расчет. Окончательный выбор требует подтверждения специалистом.
```

### Главный блок результата для покупателя

Главный блок не должен начинаться с `H` и `Q`. Он должен объяснять, под какие условия выполнен подбор.

Показать:

| Параметр | Значение |
|---|---|
| Подобрано под скважину | `глубина X м` |
| Уровень воды | `Y м` или `неизвестен, использован оценочный режим` |
| Расстояние до дома | `X м` |
| Самая высокая точка водоразбора | `1 этаж / 2 этаж / 3 этаж / другое значение` |
| Выбранные точки расхода | `раковина, душ, унитаз...` |
| Диаметр обсадной трубы | `X мм` или `неизвестен` |
| Напряжение | `220 В / 380 В` |

Если уровень воды неизвестен, рядом со строкой уровня воды показать предупреждение:

```text
Уровень воды неизвестен. Использован оценочный режим 70% от глубины скважины. Подбор требует подтверждения специалистом.
```

Если диаметр обсадной трубы неизвестен, рядом со строкой диаметра показать предупреждение:

```text
Для точного подбора нужно уточнить внутренний диаметр обсадной трубы.
```

### Раскрывающийся блок: Показать технический расчет

Технический расчет `H` и `Q` должен быть скрыт по умолчанию в раскрывающемся блоке.

Название блока:

```text
Показать технический расчет
```

Внутри показать:

| Параметр | Значение |
|---|---:|
| Уровень воды, использованный в расчете | `Y м` |
| Требуемое давление | `30 м` |
| Вертикальный подъем | `Z м` |
| Потери в трубе | `P м` |
| Требуемый напор H | `H м` |
| Требуемая подача Q | `Q л/мин` |

Пример:

```text
Использованный уровень воды: 42 м
Требуемое давление: 30 м
Вертикальный подъем: 6 м
Потери в трубе: 3 м
Требуемый напор H: 81 м
Требуемая подача Q: 38 л/мин
```

### Группы товаров

### 1. Лучшая цена

Самый дешевый товар среди подходящих.

Назначение:

```text
Самый доступный по цене вариант, который обычно подходит под введенные условия.
```

### 2. Оптимальный выбор

Баланс достаточного запаса и цены.

Для MVP:

Используется первый товар из основной сортировки:

```text
минимальный запас напора
минимальный запас подачи
затем цена
```

### 3. С запасом

Товар с большим, но разумным запасом.

Рекомендуемое условие:

```text
запас_напора >= 10 м
или
запас_подачи >= 10 л/мин
```

Не стоит показывать явно избыточный насос как предпочтительный вариант, если есть более сбалансированные товары.

### 4. Премиум / брендовый вариант

Если есть метаданные по брендам, выбрать товар из брендов с повышенным приоритетом.

Если рейтинга или приоритета брендов нет, этот блок лучше не показывать.

Рекомендуемое поведение:

```text
Показывать премиум-вариант только если есть данные о приоритете брендов.
```

Не нужно придумывать качество бренда без данных.

### Карточка товара

В каждой карточке результата показать:

- название товара;
- изображение;
- цену;
- наличие;
- максимальный напор;
- максимальную подачу;
- диаметр насоса;
- напряжение;
- запас напора;
- запас подачи;
- ссылку на страницу товара;
- кнопку добавления в корзину, если это уместно.

Осторожная формулировка рекомендации:

```text
Обычно подходит для введенных параметров.
```

Для товаров с большим запасом:

```text
Обычно подходит с дополнительным запасом.
```

Для всех карточек:

```text
Предварительный подбор. Перед покупкой подтвердите выбор со специалистом.
```

## 7. Поведение при отсутствии подходящих товаров

### Нет полностью подходящих товаров

Если ни один товар не прошел все применимые фильтры:

```text
В наличии нет насоса, который полностью соответствует расчетным параметрам.
```

Дальше можно предложить:

- показать товары с подходящим напряжением, но недостаточным напором или подачей отдельным блоком;
- если диаметр известен, предложить проверить внутренний диаметр обсадной трубы;
- предложить проверить доступное напряжение;
- предложить обратиться к специалисту.

Недостаточные по параметрам товары нельзя показывать как подходящие.

### Нет подходящих товаров из-за диаметра

Этот сценарий возможен только если диаметр обсадной трубы известен.

Если товары подходят по `H`, `Q` и напряжению, но не подходят по диаметру:

```text
Насос с нужными параметрами может не поместиться в выбранную обсадную трубу.
```

Показать:

```text
Проверьте внутренний диаметр обсадной трубы, а не только ее номинальный размер.
```

### Диаметр обсадной трубы неизвестен

Если пользователь выбрал `не знаю`, подбор продолжается без жесткого фильтра по диаметру.

Показать:

```text
Диаметр обсадной трубы не указан. Насосы обычно подходят по расчетному напору и подаче, но совместимость по диаметру требует уточнения.
```

### Нет подходящих товаров из-за напряжения

Если товары подходят по `H`, `Q` и, при известном диаметре, по диаметру, но не подходят по напряжению:

```text
Для выбранного напряжения сейчас нет подходящего насоса в наличии.
```

Дополнительно можно показать:

```text
Подходящие насосы есть в варианте 380 В.
```

Показывать это сообщение нужно только если оно действительно соответствует данным.

### Нет подходящих товаров из-за наличия

Если обычно подходящие товары есть в каталоге, но `stock = 0`:

```text
В каталоге есть обычно подходящие насосы, но сейчас они отсутствуют на складе.
```

Вместо рекомендации к покупке лучше показать контакт с менеджером или специалистом.

### Неизвестный уровень воды

Если уровень воды неизвестен, расчет разрешен, но результат должен быть явно помечен:

```text
Ориентировочный результат: уровень воды был оценен как 70% от общей глубины скважины.
```

Нельзя использовать категоричные формулировки.

## 8. Заметки по реализации для OpenCart 2.3

### Рекомендуемая архитектура

Создать отдельный модуль, например:

```text
catalog/controller/extension/module/pump_selector.php
catalog/model/extension/module/pump_selector.php
catalog/view/theme/default/template/extension/module/pump_selector.tpl
```

В зависимости от темы путь к шаблону может отличаться.

На этом этапе не нужно писать production-код модуля, делать frontend или AJAX. Перед разработкой модуля нужно подготовить техническую основу БД.

### Источник данных

Так как в MVP есть около 90 подходящих товаров, лучше использовать отдельный нормализованный источник товарных параметров.

Для MVP явно лучше использовать отдельную нормализованную таблицу товаров, а не парсить атрибуты OpenCart на лету во время каждого пользовательского подбора.

### Ответственность контроллера в будущей реализации

Контроллер должен:

1. отрисовать форму селектора;
2. валидировать ввод;
3. нормализовать пользовательские значения;
4. вызвать расчетную логику;
5. вызвать логику фильтрации товаров;
6. вернуть страницу результата.

### Ответственность модели в будущей реализации

Модель должна:

1. загрузить товары из `oc_pump_selector_product`;
2. связать их с `oc_product`;
3. применить фильтры;
4. рассчитать запасы;
5. отсортировать товары;
6. вернуть сгруппированных кандидатов для экрана результата.

### Важные ограничения MVP

Нужно явно указывать:

```text
Этот селектор выполняет упрощенный предварительный подбор скважинного насоса.
Он не заменяет профессиональный гидравлический расчет.
Окончательный выбор насоса требует подтверждения специалистом.
```

Не нужно утверждать:

- гарантированную совместимость;
- точные гидравлические характеристики системы;
- точное давление в каждой точке водоразбора;
- пригодность для всех диаметров и схем трубопровода;
- подбор только по общей глубине скважины.

| Остаток товара | Больше 0 |
| Цена товара | Больше 0 |

## 9. Следующий технический шаг: нормализованная таблица товаров

Перед началом разработки модуля нужно подготовить таблицу `oc_pump_selector_product`. Она должна хранить уже очищенные числовые параметры насосов, чтобы селектор не парсил текстовые атрибуты OpenCart на лету.

В production-архитектуре `oc_pump_selector_product` считается нормализованным кешем данных для подборщика, а не источником правды.

Источник правды:

```text
oc_product
oc_product_attribute
oc_attribute_description
oc_manufacturer
oc_product_to_category
```

Ручной запуск seed SQL допустим только для тестирования. В рабочей реализации кеш должен обновляться отдельным backend-действием, а не при каждом пользовательском подборе.

### Назначение таблицы

Таблица хранит только параметры, нужные для предварительного подбора:

- максимальный напор;
- максимальную подачу;
- диаметр насоса;
- напряжение;
- признак участия в селекторе;
- служебные поля для повторной нормализации.

Цена, остаток, статус товара, название, изображение, производитель и ссылка на карточку остаются в стандартных таблицах OpenCart.

### Структура таблицы

Рекомендуемая таблица:

```text
oc_pump_selector_product
```

Поля:

| Поле | Тип данных | Назначение |
|---|---|---|
| `product_id` | `INT(11) NOT NULL` | ID товара из `oc_product.product_id` |
| `max_head_m` | `DECIMAL(8,2) NOT NULL` | Максимальный напор, м |
| `max_flow_l_min` | `DECIMAL(8,2) NOT NULL` | Максимальная подача, л/мин |
| `pump_diameter_mm` | `DECIMAL(8,2) NOT NULL` | Диаметр насоса, мм |
| `voltage` | `VARCHAR(20) NOT NULL` | Нормализованное напряжение: `220` или `380` |
| `brand_priority` | `TINYINT(3) UNSIGNED DEFAULT NULL` | Приоритет бренда для премиум-блока, если используется |
| `is_eligible` | `TINYINT(1) NOT NULL DEFAULT 1` | Товар участвует в селекторе |
| `source_hash` | `VARCHAR(64) DEFAULT NULL` | Хеш исходных атрибутов для контроля повторной нормализации |
| `normalized_at` | `DATETIME DEFAULT NULL` | Когда параметры были нормализованы |
| `updated_at` | `DATETIME DEFAULT NULL` | Когда запись была обновлена вручную или повторно |

### Ключи и индексы

Рекомендуемые ключи:

```text
PRIMARY KEY (product_id)
```

Рекомендуемые индексы:

```text
INDEX idx_pump_selector_filter (is_eligible, voltage, max_head_m, max_flow_l_min)
INDEX idx_pump_selector_diameter (pump_diameter_mm)
INDEX idx_pump_selector_brand_priority (brand_priority)
```

Пояснение:

- `PRIMARY KEY (product_id)` гарантирует одну нормализованную запись на товар;
- `idx_pump_selector_filter` помогает базовой фильтрации по доступности для селектора, напряжению, напору и подаче;
- `idx_pump_selector_diameter` нужен для фильтра по диаметру, когда пользователь его указал;
- `idx_pump_selector_brand_priority` нужен только если используется блок `Премиум / брендовый вариант`.

### Связь с `oc_product`

Связь выполняется по `product_id`:

```text
oc_pump_selector_product.product_id = oc_product.product_id
```

В выборке товаров нужно дополнительно учитывать стандартные поля OpenCart:

```text
oc_product.product_id
oc_product.price > 0
oc_product.quantity > 0
oc_product.status = 1
```

Если в магазине используется другая логика наличия, например статусы склада или опции, это нужно учесть отдельно в будущей реализации.

Для вывода карточек также понадобятся стандартные связи OpenCart:

- `oc_product_description` для названия;
- `oc_product.image` для изображения;
- `oc_manufacturer` для бренда, если нужен премиум-блок;
- SEO URL или стандартный маршрут товара для ссылки.

### Как наполнять и обновлять кеш из существующих атрибутов товаров

Наполнение и повторное обновление таблицы должно выполняться отдельным backend-методом:

```text
rebuildSelectorProducts()
```

Метод пересобирает нормализованный кеш. Он не должен запускаться внутри пользовательского подбора.

Процесс:

1. Очистить `oc_pump_selector_product`.
2. Найти товары из категорий MVP:
   - `11900308`;
   - `11900309`.
3. Прочитать существующие атрибуты OpenCart из `oc_product_attribute` и `oc_attribute_description`.
3. Сопоставить атрибуты с нужными полями:
   - максимальный напор -> `max_head_m`;
   - максимальная подача -> `max_flow_l_min`;
   - диаметр насоса -> `pump_diameter_mm`;
   - напряжение -> `voltage`.
4. Проверить обязательные характеристики:
   - `Максимальный напор`;
   - `Максимальная производительность`;
   - `Диаметр насоса`.
5. Очистить значения от единиц измерения и привести к числам.
6. Для `max_flow_l_min` извлечь именно значение в `л/мин`, например из строки `1,8 куб/час (30л/мин)` получить `30`.
7. Привести напряжение к одному формату: `220` или `380`.
8. Получить производителя из `oc_manufacturer` и выставить `brand_priority`:
   - `Pedrollo%` -> `10`;
   - `Belamos%` -> `5`;
   - остальные -> `0`.
9. Записать только товары, у которых:
   - `oc_product.price > 0`;
   - `oc_product.quantity > 0`;
   - `oc_product.status = 1`.
10. Установить `is_eligible = 1` для записанных товаров.
11. Для товаров с неполными или сомнительными данными не добавлять запись в кеш до ручной проверки.

Примеры нормализации:

```text
"90 м" -> 90.00
"90m" -> 90.00
"90" -> 90.00
"3\"" -> 76.20 или вручную уточненное значение в мм
"220 В" -> "220"
"1 фазный / 220В" -> "220"
"380 В" -> "380"
```

Значения с неоднозначными единицами лучше отправлять на ручную проверку, а не автоматически считать корректными.

Метод `rebuildSelectorProducts()` должен возвращать статистику:

```text
total_scanned
eligible_inserted
skipped_no_attributes
skipped_no_price
skipped_no_stock
skipped_disabled
```

Для MVP достаточно текущих двух категорий. Архитектуру нужно оставить расширяемой, чтобы позже можно было добавить новые категории, бренды и правила приоритета без переписывания пользовательского подбора.

### Способы запуска обновления кеша

Рекомендуемые способы запуска `rebuildSelectorProducts()`:

- кнопка в админке `Обновить данные подборщика`;
- protected cron URL с секретным `token`.

Админская кнопка нужна для ручного обновления после изменения карточек товаров или характеристик.

Cron URL нужен для автоматической периодической пересборки, например после синхронизации каталога.

Protected cron URL должен проверять секретный `token`. Без корректного token пересборка не должна запускаться.

Важно:

- не пересобирать `oc_pump_selector_product` при каждом пользовательском подборе;
- пользовательский подбор должен только читать готовый кеш;
- пересборка кеша должна быть отдельной служебной операцией;
- ручные seed SQL-файлы остаются тестовым инструментом, а не production-механизмом.

### Как обновлять данные вручную или повторно

Нужно предусмотреть два режима обновления данных:

1. Ручное редактирование нормализованной записи.
2. Повторная нормализация из атрибутов товара.

Ручное редактирование нужно для случаев, когда:

- атрибуты товара заполнены нестандартно;
- диаметр указан в дюймах и требует проверки;
- напряжение описано текстом;
- брендовый приоритет задается вручную;
- товар временно нужно исключить из селектора.

Повторная нормализация нужна, когда:

- изменились исходные атрибуты товара;
- добавлены новые насосы;
- исправлены ошибки в карточках товаров;
- нужно пересобрать таблицу после изменения правил нормализации.

При повторной нормализации желательно:

- обновлять `source_hash`;
- обновлять `normalized_at`;
- не затирать вручную исправленные значения без явного подтверждения;
- вести список товаров, которые не прошли нормализацию.

### Какие товары считаются `eligible`

Товар считается подходящим для участия в селекторе, если выполняются все условия:

- это погружной скважинный насос;
- товар активен в OpenCart: `oc_product.status = 1`;
- у товара есть нормализованный `max_head_m > 0`;
- у товара есть нормализованный `max_flow_l_min > 0`;
- у товара есть нормализованный `pump_diameter_mm > 0`;
- у товара есть нормализованное напряжение `220` или `380`;
- товар не исключен вручную из селектора;
- товар относится к набору товаров, который бизнес считает допустимым для MVP.

В MVP `rebuildSelectorProducts()` записывает в кеш только товары с актуальными `price > 0`, `quantity > 0`, `status = 1`. При этом цена, остаток и статус меняются чаще атрибутов, поэтому их нужно дополнительно проверять при каждом подборе:

```text
price > 0
stock > 0
status = 1
```

В OpenCart 2.3 остаток обычно хранится в `oc_product.quantity`. В спецификации селектора допустимо использовать термин `stock`, но при реализации нужно связать его с фактической логикой наличия магазина.

## Основной принцип MVP

Селектор должен отвечать на вопрос:

```text
Какие скважинные насосы в наличии обычно подходят по требуемому напору, требуемой подаче, диаметру обсадной трубы и напряжению?
```

Если диаметр обсадной трубы неизвестен, вопрос меняется:

```text
Какие скважинные насосы в наличии обычно подходят по требуемому напору, требуемой подаче и напряжению, но требуют уточнения совместимости по диаметру?
```

Селектор не должен отвечать на вопрос:

```text
Какой насос гарантированно идеально подходит для этой скважины?
```
