# AI-ASSISTED STANDARDIZATION — ИДЕЯ НОВОГО ПРОЕКТА

Дата фиксации идеи: 14-07-2026

Статус: `concept / implementation not started`

## 1. Назначение документа

Этот документ фиксирует идею нового проекта по стандартизации характеристик товарного каталога с использованием:

- Python как детерминированного ядра обработки данных;
- ChatGPT как семантического аналитического компонента;
- человека как обязательного участника принятия решений и разрешения изменений.

Проект рассматривается как отдельное направление.

Он не является автоматическим продолжением, обновлением или переписыванием существующего `framework-standardization`.

Разработка нового проекта не начата.

Документ фиксирует только концепцию, возникшую после практического опыта автоматизированной и ручной стандартизации характеристик сайта Home-Energetika.

---

## 2. Предпосылки появления идеи

Первоначально стандартизацию характеристик предполагалось реализовать в виде PHP-фреймворка.

В ходе проектирования были предусмотрены:

- реестр канонических атрибутов;
- контракты характеристик;
- нормализаторы;
- DB-readonly аудит;
- preview;
- apply-plan;
- migration;
- cleanup;
- runtime-проверки;
- обработка конфликтов;
- идемпотентность;
- документация;
- handoff между рабочими сессиями.

Создание `AUTOMATION_ROADMAP.md` показало, что полноценная реализация такого подхода представляет собой большой самостоятельный инженерный проект.

Одновременно срочная ручная работа непосредственно с продуктивной базой показала другой эффективный цикл:

`инвентаризация данных → семантический анализ → решение человека → SQL → проверка → журнал`

В этом процессе наиболее трудоёмкая механическая работа может быть передана Python, а наиболее сложная смысловая часть — выполняться с помощью ChatGPT.

### 2.1 Происхождение концепции

Идея сочетания Python, обычного ChatGPT и ручного подтверждения фактически соответствует первоначальному представлению об автоматизации стандартизации.

Изначально предполагалось не создание полностью автономного фреймворка, а процесс, в котором:

- программа выполняет механический анализ данных;
- ChatGPT помогает интерпретировать характеристики и исключения;
- человек принимает окончательные решения;
- изменения выполняются только после проверки.

На раннем этапе эта идея не была достаточно ясно сформулирована, поэтому проектирование сместилось в сторону универсального PHP-фреймворка, который должен был заранее формализовать большую часть предметной области.

Работа над `framework-standardization` позволила выявить необходимые ограничения, проверки и механизмы безопасности. После оценки реального масштаба задачи концепция вернулась к первоначальному замыслу, но уже в более точной и практически реализуемой форме.

## 3. Основная проблема

Характеристики товаров невозможно надёжно стандартизировать только по названиям атрибутов и простым правилам замены.

В данных встречаются:

- дублирующиеся названия;
- разные характеристики с похожими названиями;
- числа с единицами измерения;
- диапазоны;
- приблизительные значения;
- несколько вариантов в одной строке;
- разные системы единиц;
- неверно заполненные поля;
- значения другой характеристики;
- текст вместо числа;
- противоречия между названием, атрибутами и описанием;
- данные, зависящие от конкретной модели товара;
- исключения, требующие проверки технической документации.

Примеры смысловых различий:

- максимальная, минимальная и номинальная производительность;
- максимальный, минимальный и номинальный напор;
- диаметр насоса, подключения, резьбы, трубы и скважины;
- мощность двигателя, потребляемая и номинальная мощность;
- максимальная и минимальная температура жидкости.

Поэтому полностью универсальная детерминированная система требует большого количества заранее созданных контрактов и специальных правил.

---

## 4. Основная гипотеза нового проекта

Новый проект строится на следующем разделении ответственности:

> Python выполняет всё, что должно быть точным, проверяемым и воспроизводимым.  
> ChatGPT анализирует смысл, классифицирует неоднозначные данные и предлагает решения.  
> Человек подтверждает потенциально опасные изменения.

Это не система, в которой искусственный интеллект самостоятельно изменяет продуктивную базу.

Это управляемый конвейер, в котором:

- известное обрабатывается детерминированно;
- новое и неоднозначное анализируется моделью;
- все предложения проверяются Python;
- опасные случаи блокируются;
- запись разрешается человеком;
- результат фиксируется в журнале.

## 5. Концептуальное отличие от прежнего подхода

### Прежний подход

Сначала предполагалось максимально полно описать предметную область:

- создать контракты;
- зарегистрировать aliases;
- реализовать нормализаторы;
- определить все допустимые значения;
- описать исключения;
- только после этого запускать обработку.

Фактически система должна была заранее знать почти все возможные варианты данных.

### Новый подход

Новый проект должен развиваться от реальных данных:

1. Python исследует выбранную область каталога.
2. Система выделяет кандидатов и аномалии.
3. ChatGPT предлагает смысловую классификацию.
4. Python проверяет предложение.
5. Человек подтверждает решение.
6. Подтверждённое решение сохраняется как контракт.
7. При следующих запусках уже известные случаи обрабатываются без повторного анализа.

Таким образом, система накапливает знания постепенно.

## 6. Предполагаемая архитектура

### 6.1. Extractor

Отвечает за безопасное получение контролируемого снимка данных.

Функции:

- чтение SQL dump;
- чтение локальной копии базы;
- ограничение по категории;
- ограничение по языку;
- выборка товаров и атрибутов;
- сохранение исходного состояния;
- создание воспроизводимого dataset snapshot.

Extractor не должен изменять продуктивные данные.

### 6.2. Profiler

Выполняет механический анализ данных.

Функции:

- подсчёт товаров и строк;
- группировка по атрибутам;
- распределение значений;
- поиск одинаковых названий;
- поиск похожих названий;
- поиск пересечений по товарам;
- определение canonical-only, alias-only и both;
- поиск пустых значений;
- классификация чисел, диапазонов и текста;
- формирование списка потенциальных конфликтов;
- подготовка компактного пакета для анализа.

### 6.3. Contract Registry

Хранит уже подтверждённые знания.

Контракт характеристики может содержать:

- смысловой ключ;
- канонический `attribute_id`;
- каноническое название;
- единицу измерения;
- тип значения;
- известные aliases;
- допустимые преобразования;
- запрещённые соседние характеристики;
- допустимые диапазоны;
- известные исключения;
- правила итоговой проверки;
- историю подтверждения.

Контракт появляется после подтверждения решения, а не обязательно до первого анализа характеристики.

### 6.4. Deterministic Normalizers

Обрабатывают уже известные безопасные форматы.

Примеры:

- удаление единицы измерения;
- замена десятичной запятой на точку;
- перевод лет в месяцы;
- перевод `м³/ч` в `л/мин`;
- нормализация `IP 68 → IP68`;
- нормализация `220 В → 220`;
- нормализация булевых значений.

Нормализатор не должен принимать семантические решения за пределами зарегистрированного контракта.

### 6.5. AI Analyst

Получает не полный dump, а подготовленный компактный пакет данных.

Задачи:

- определить, являются ли атрибуты смысловыми дублями;
- предложить каноническое название;
- классифицировать неоднозначные значения;
- выявить характеристики, которые нельзя объединять;
- предложить безопасные преобразования;
- выделить строки для ручной проверки;
- объяснить основание каждого решения;
- оценить уверенность.

AI Analyst должен возвращать структурированный результат, а не произвольный текст.

### 6.6. Validator

Проверяет каждое предложение AI Analyst.

Возможные проверки:

- товар существует;
- товар входит в разрешённую область;
- исходный атрибут существует;
- исходное значение не изменилось после анализа;
- целевой атрибут существует;
- у товара нет конфликтующей канонической строки;
- предлагаемое значение соответствует типу контракта;
- преобразование разрешено;
- количество строк совпадает с preview;
- запрос не затрагивает данные вне области;
- в плане отсутствуют неожиданные `DELETE`;
- результат повторного запуска идемпотентен.

Невалидное предложение не допускается до SQL-плана.

### 6.7. Review Queue

Содержит случаи, которые нельзя обработать автоматически.

Для каждого случая сохраняются:

- `product_id`;
- название товара;
- модель;
- исходный атрибут;
- исходное значение;
- предполагаемый канон;
- причина неоднозначности;
- предложение модели;
- уровень уверенности;
- решение человека;
- источник подтверждения.

### 6.8. Plan Generator

Формирует детерминированный план изменений.

План должен содержать:

- ожидаемое количество строк;
- список товаров;
- старое значение;
- новое значение;
- источник решения;
- тип операции;
- precondition;
- postcondition;
- rollback information.

### 6.9. Apply

Запись выполняется только после отдельного подтверждения человека.

Возможные режимы:

- генерация SQL без выполнения;
- выполнение на локальной копии;
- выполнение на staging;
- выполнение на production после подтверждения.

Для первой версии предпочтителен режим генерации SQL без самостоятельной записи в production.

### 6.10. Verify

После применения система повторяет аудит.

Проверяются:

- количество строк канона;
- количество товаров;
- остатки aliases;
- конфликты;
- неподдерживаемые значения;
- пустые значения;
- соответствие ожидаемому результату;
- идемпотентность повторного запуска.

### 6.11. Audit Log

Каждое подтверждённое изменение фиксируется.

Запись должна содержать:

- дату;
- область;
- характеристику;
- исходное состояние;
- принятое решение;
- применённые преобразования;
- исключения;
- количество изменённых строк;
- итоговую проверку;
- пользователя, подтвердившего apply;
- версию контракта;
- версию модели или режим анализа.

## 7. Предполагаемый рабочий поток

Общий процесс:

`OpenCart / SQL dump`

`→ Extractor`

`→ Profiler`

`→ существующий контракт и детерминированные правила`

`→ AI Analyst для неизвестных случаев`

`→ Validator`

`→ Review Queue`

`→ подтверждение человека`

`→ Plan Generator`

`→ SQL preview`

`→ apply`

`→ Verify`

`→ Audit Log`

Известные случаи не должны каждый раз повторно отправляться ChatGPT.

AI-компонент подключается только тогда, когда:

- отсутствует контракт;
- нормализатор не распознал значение;
- найден конфликт;
- значение неоднозначно;
- требуется смысловое решение.

## 8. Формат взаимодействия с ChatGPT

AI Analyst должен получать минимально достаточный контекст.

Пример входного пакета:

- название предполагаемой характеристики;
- список атрибутов-кандидатов;
- количество товаров;
- распределение значений;
- пересечения;
- несколько репрезентативных карточек;
- модель товара;
- описание только для конфликтных случаев;
- известные запреты;
- существующий контракт, если он есть.

Пример структурированного результата:

- канонический атрибут;
- каноническое название;
- aliases;
- отдельные смысловые характеристики;
- безопасные преобразования;
- строки с высокой уверенностью;
- строки для ручной проверки;
- запрещённые операции;
- объяснение;
- confidence;
- требуемые источники проверки.

Свободный текст модели не должен напрямую превращаться в SQL.

## 9. Два возможных этапа реализации

### 9.1. Этап 1 — Python + текущий ChatGPT

На первом этапе интеграция с API не обязательна.

Python формирует файлы:

- `attribute_summary.json`;
- `value_distribution.csv`;
- `overlaps.csv`;
- `conflicts.csv`;
- `review_queue.csv`;
- `analysis_packet.md`.

Пакет передаётся в обычный чат ChatGPT.

Результат сохраняется в структурированном JSON или YAML и передаётся Validator.

Преимущества:

- минимальная стоимость разработки;
- отсутствие расходов на API;
- быстрое тестирование формата;
- возможность изменить процесс до автоматизации;
- сохранение обязательного человеческого контроля.

### 9.2. Этап 2 — Python + OpenAI API

После стабилизации форматов можно подключить API.

Требования перед переходом:

- устойчивая входная схема;
- устойчивая схема ответа;
- проверенные валидаторы;
- набор реальных тестовых случаев;
- ограничения стоимости;
- ограничения размера контекста;
- retry policy;
- журналирование запросов;
- защита от повторного apply;
- механизм ручной эскалации.

---

### 9.3. Зафиксированное решение по OpenAI API и масштабу проекта

На текущем этапе OpenAI API не является обязательной частью проекта и не входит в предполагаемый MVP.

Основная причина — стандартизация характеристик Home-Energetika рассматривается преимущественно как ограниченная, возможно одноразовая работа. Создание полноценной API-интеграции, агентного контура или автономной AI-системы может потребовать больше ресурсов, чем даст экономии в рамках текущей задачи.

Базовый рабочий вариант:

`Python → обычный ChatGPT → структурированное решение → Python Validator → SQL preview → ручное применение → post-check`

Распределение ролей:

- Python извлекает и профилирует данные;
- Python формирует компактный пакет для анализа;
- пользователь вручную передаёт пакет в обычный ChatGPT;
- ChatGPT выполняет смысловой анализ и предлагает решение;
- решение сохраняется в JSON, YAML, CSV или другом согласованном формате;
- Python проверяет решение;
- Python формирует SQL preview и контрольные запросы;
- пользователь подтверждает и вручную применяет изменения;
- Python выполняет итоговую проверку.

Таким образом, обычный ChatGPT используется как внешний семантический помощник без программной интеграции.

OpenAI API может быть рассмотрен позднее только при выполнении хотя бы одного из условий:

- стандартизацию потребуется повторять для многих категорий;
- аналогичная работа появится на других сайтах;
- будет накоплен устойчивый формат входных пакетов;
- будет сформирована стабильная схема структурированного ответа;
- ручная передача данных между Python и ChatGPT станет заметным узким местом;
- автоматизация через API даст измеримую экономию времени и средств.

До появления таких условий не следует:

- разрабатывать API-клиент;
- строить автономного агента;
- проектировать очередь AI-заданий;
- создавать SaaS;
- реализовывать автоматический AI-to-production;
- расширять проект до универсального продукта.

Уточнённая рабочая формулировка:

> Ограниченный Python-инструментарий для ускорения ручной стандартизации характеристик с использованием обычного ChatGPT как внешнего семантического помощника.

Это уточнение имеет приоритет над формулировками документа, которые могут создавать впечатление, что OpenAI API является обязательной частью MVP или конечной архитектуры.

## 10. Человеческое подтверждение

Человек остаётся обязательным участником процесса.

Без подтверждения не должны выполняться:

- изменение канонического атрибута;
- объединение новых aliases;
- массовое преобразование нового типа;
- удаление строк;
- разрешение конфликтов;
- применение плана к production;
- изменение данных вне утверждённой категории;
- изменение глобального справочника атрибутов.

Возможное разделение уровней:

- `AUTO_SAFE` — известное правило, прошедшее все проверки;
- `REVIEW_REQUIRED` — требуется подтверждение;
- `BLOCKED` — предложение нарушает контракт или безопасность;
- `UNSUPPORTED` — система не умеет обработать случай;
- `RESEARCH_REQUIRED` — требуется внешний источник или техническая документация.

## 11. Безопасность

Минимальные требования:

- production по умолчанию недоступен для записи;
- обязательный scope по категории;
- обязательный `language_id`;
- снимок исходных значений;
- precondition для каждой строки;
- точное ожидаемое количество изменений;
- запрет широких запросов без списка товаров или условий;
- запрет неконтролируемого глобального удаления атрибутов;
- обязательный preview;
- отдельное подтверждение apply;
- post-check;
- rollback-план;
- журнал всех решений.

AI-компонент не должен иметь права самостоятельно расширять область операции.

## 14.1 Практическая ценность работы над PHP-фреймворком

Переход к ограниченному Python-инструментарию и обычному ChatGPT не означает, что работа над архитектурой `framework-standardization` была напрасной.

Полноценный PHP-фреймворк оказался избыточным для срочной разовой очистки уже накопленных данных. Однако разработанные в нём концепции имеют прямую практическую ценность для следующей задачи — доработки импорта OpenCart.

После завершения ручной стандартизации необходимо изменить импорт так, чтобы он больше не создавал новые подмножества, дубли и вариации одних и тех же характеристик.

Для этой задачи особенно полезны следующие идеи `framework-standardization`:

- реестр канонических атрибутов;
- явное сопоставление входных характеристик с канонами;
- registry известных aliases;
- нормализация названий и значений до записи в базу;
- проверка смысловой совместимости характеристик;
- запрет автоматического создания нового атрибута при неоднозначном совпадении;
- разделение safe, conflict, unsupported и review-required случаев;
- идемпотентность повторного импорта;
- предварительный preview;
- runtime-проверки;
- аудит принятых решений;
- контролируемая обработка исключений.

Таким образом, две задачи дополняют друг друга:

1. Ручная стандартизация устраняет уже накопленный исторический беспорядок.
2. Доработанный импорт предотвращает повторное появление такого беспорядка.

Архитектурные наработки PHP-фреймворка могут быть использованы не как отдельная универсальная система очистки каталога, а как основа защитного слоя импорта.

Возможная будущая схема:

`входные данные поставщика → распознавание канона → нормализация → валидация → запись в существующий атрибут`

При отсутствии надёжного соответствия импорт не должен самостоятельно создавать новую характеристику.

Такой случай должен попадать в отдельный отчёт или очередь ручного разбора.

Следовательно, работа над `framework-standardization` дала два результата:

- показала границы экономически оправданной автоматизации очистки;
- сформировала архитектурные принципы, необходимые для безопасной доработки импорта OpenCart.

## 19. Основные риски

### 19.1. Ошибки AI-анализа

Модель может:

- ошибочно объединить разные характеристики;
- придумать недостающий факт;
- неверно интерпретировать модель товара;
- переоценить уверенность;
- предложить неподтверждённое значение.

Меры:

- структурированный ответ;
- обязательный Validator;
- confidence не является разрешением;
- блокировка неподтверждённых решений;
- ручная проверка;
- сохранение исходных данных.

### 19.2. Избыточный контекст и стоимость

Передача полного каталога модели будет дорогой и неэффективной.

Меры:

- предварительное профилирование Python;
- агрегация значений;
- выбор репрезентативных примеров;
- отправка только конфликтных строк;
- использование накопленных контрактов;
- кэширование подтверждённых решений.

### 19.3. Непредсказуемость результата

Одинаковый запрос может давать немного разные объяснения.

Меры:

- строгая схема результата;
- детерминированные валидаторы;
- сохранение решения;
- повторное использование контракта;
- тесты на реальных fixtures.

### 19.4. Смешение автоматизации и ручной работы

Существует риск считать предложение модели готовым изменением.

Меры:

- чёткие состояния;
- отдельный review queue;
- отдельное подтверждение;
- запрет прямого AI-to-production.

### 19.5. Повторное разрастание проекта

Новый проект может снова превратиться в чрезмерно большую систему.

Меры:

- узкий MVP;
- одна категория;
- одна характеристика;
- отсутствие production apply;
- фиксированные критерии завершения;
- запрет расширения scope до подтверждения MVP.
