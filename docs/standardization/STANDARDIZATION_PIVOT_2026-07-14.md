# STANDARDIZATION PIVOT — 14-07-2026

## 1. Назначение документа

Этот документ фиксирует ключевой поворот в работе над стандартизацией характеристик товаров сайта Home-Energetika, произошедший 14-07-2026.

В течение дня первоначальный план автоматизации был переоценён. После уточнения реального масштаба задачи было принято решение:

- временно законсервировать разработку `framework-standardization`;
- перейти к срочной ручной обработке данных непосредственно в продуктивной базе OpenCart;
- использовать Codex экономно — только для ограниченных инженерных задач, где его применение действительно оправдано.

Документ не является `HANDOFF.md` фреймворка и не описывает его текущее инженерное состояние.

Он относится ко всему проекту Home-Energetika и фиксирует:

- причины изменения стратегии;
- новый рабочий режим;
- границы между фреймворком и ручной обработкой;
- фактические результаты дня;
- предварительную оценку оставшегося объёма.

## 4. Причины переоценки подхода

При дальнейшем анализе выяснилось, что характеристики отличаются не только названиями и единицами измерения.

Одна и та же смысловая характеристика может содержать:

- простые числовые значения;
- числа с единицами;
- диапазоны;
- приблизительные значения;
- несколько значений в одной строке;
- характеристики нескольких вариантов товара;
- ошибочно записанные значения другой характеристики;
- текст вместо числа;
- данные, требующие проверки по описанию карточки;
- данные, требующие проверки по конкретной модели производителя.

Кроме того, похожие или одинаковые по названию атрибуты могут иметь разный смысл.

Примеры:

- максимальная, минимальная, номинальная и обычная производительность;
- максимальный, минимальный и номинальный напор;
- диаметр насоса, диаметр подключения, диаметр резьбы, диаметр трубы и диаметр скважины;
- мощность двигателя, потребляемая мощность и номинальная мощность;
- максимальная и минимальная температура жидкости.

Следовательно, универсальное правило объединения по похожему названию применять нельзя.

## 5. Концепция экономного использования Codex

В ходе работы была сформулирована концепция экономного использования Codex.

Codex имеет смысл применять там, где результатом являются ограниченные и проверяемые изменения в репозитории:

- реализация конкретной функции;
- изменение CLI-команды;
- добавление нормализатора;
- написание тестов;
- подготовка runtime-проверки;
- документирование уже принятого решения;
- исправление конкретного дефекта.

Codex невыгодно использовать для длинного ручного исследования каждой характеристики, когда основная работа состоит из:

- анализа строк базы данных;
- сравнения карточек товаров;
- уточнения значений отдельных моделей;
- последовательного выполнения большого количества SQL-запросов;
- принятия решений по одному исключению за раз.

Принятый принцип:

> Codex используется как инженерный инструмент, а не как дорогая оболочка для ручного исследования данных.

`framework-standardization/docs/HANDOFF.md`

Его содержимое не обновлялось результатами ручной работы, поскольку ручная продуктивная обработка является отдельным рабочим контуром и не относится к разработке фреймворка.

## 7. Переход к срочной ручной обработке

После постановки фреймворка на паузу работа была переведена в режим срочной ручной стандартизации.

### 7.1. Рабочая база

Изменения выполняются непосредственно в продуктивной базе OpenCart через phpMyAdmin.

Отдельный режим только для чтения для текущей ручной задачи не используется.

### 7.2. Распределение ответственности

- SQL подготавливает ChatGPT.
- SQL вручную выполняет пользователь.
- ChatGPT не подключается к продуктивной базе.
- ChatGPT не выполняет запросы самостоятельно.
- Пользователь сообщает фактический результат каждого запроса.
- Следующее действие определяется только после проверки результата предыдущего.

### 7.3. Основное правило

> Строго один SQL-запрос за раз.

Следующий запрос подготавливается только после получения и проверки результата предыдущего.

Пакеты SQL-запросов в текущем режиме не используются.

Ранее полученный иерархический охват с подкатегориями к текущей ручной задаче не относится и не используется для продуктивных изменений.

## 9. Принципы ручной стандартизации

Для каждой смысловой характеристики выполняется отдельный цикл.

### 9.1. Аудит

Определяются:

- канонический атрибут;
- возможные дубли;
- количество строк;
- количество товаров;
- пересечения между атрибутами;
- распределение значений;
- ошибочные строки;
- неоднозначные строки;
- потенциальные конфликты.

### 9.2. Разделение по смыслу

Похожие по названию, но разные по смыслу характеристики не объединяются.

Перед переносом значений необходимо подтвердить, что атрибуты действительно описывают одну и ту же характеристику.

### 9.3. Каноническое название

Используется формат:

`Название характеристики (единица измерения)`

Примеры:

- `Вес (кг)`
- `Напряжение (В)`
- `Максимальный напор (м)`
- `Максимальная производительность (л/мин)`
- `Максимальная температура жидкости (°C)`

### 9.4. Формат значений

В значении хранится только само значение, без единицы измерения и пояснительного текста.

Примеры:

- `35 °C → 35`
- `220 В → 220`
- `60 л/мин → 60`
- `100 м → 100`

Предпочтительный формат числовой характеристики — одно число.

### 9.5. Работа с дублями

Дублирующий атрибут не удаляется из глобального справочника OpenCart.

Переносятся только его значения у товаров текущей категории.

Данные других категорий не изменяются.

### 9.6. Неоднозначные значения

Если значение нельзя определить надёжно, оно не изменяется автоматически.

Допускается проверка:

- остальных характеристик товара;
- описания;
- названия модели;
- артикула;
- маркировки производителя;
- официальной технической документации.

Предположение не считается достаточным основанием для продуктивного изменения.

### 9.7. Контроль результата

После завершения каждой характеристики выполняется итоговая проверка:

- количество строк канона;
- количество товаров;
- остатки старых атрибутов;
- наличие нечисловых или неподдерживаемых значений;
- наличие пересечений;
- наличие конфликтов.

Production log содержит по каждой характеристике:

- канон и дубли;
- выполненные изменения;
- обработанные исключения;
- итоговые показатели;
- то, что намеренно не изменялось.

Production log является источником истории ручных продуктивных изменений.

Этот документ не заменяет production log и не должен содержать полный перечень всех SQL-запросов.

## 14. Оценка выполненного объёма

Подсчёт только по исходным `72` атрибутам не отражает реальную трудоёмкость.

Несколько исходных атрибутов могут относиться к одной смысловой характеристике, а один сложный атрибут может потребовать больше времени, чем несколько простых.

Предварительная оценка общего объёма:

- ориентировочно `30–40` смысловых групп;
- полностью завершено `9` смысловых групп.

Оценка прогресса:

- по количеству смысловых групп: примерно `23–30%`;
- по общей трудоёмкости: примерно `30–35%`.

Рабочая итоговая оценка:

- выполнено около `30%`;
- остаётся около `70%`.

Эта оценка является предварительной и будет уточняться по мере аудита оставшихся характеристик.

Результаты ручной обработки не должны автоматически переноситься в состояние фреймворка без отдельного решения о возобновлении его разработки.

## 18. Ключевые выводы дня

### 18.1. Рабочий алгоритм ещё не является готовой автоматизацией

Успешная обработка одной группы товаров не гарантирует, что остальные характеристики имеют такую же структуру данных.

### 18.2. Сначала нужно измерить масштаб

До начала автоматизации необходимо получить реальную картину:

- число товаров;
- число атрибутов;
- число дублей;
- типы значений;
- количество исключений;
- количество конфликтов.

### 18.3. Не вся автоматизация окупается сразу

Иногда ручная обработка продуктивной базы быстрее и экономически оправданнее, чем разработка универсальной системы.

### 18.4. Автоматизация остаётся полезной

Фреймворк может быть продолжен позже, когда:

- срочная ручная задача будет завершена;
- накопится фактический опыт по всем типам характеристик;
- будут известны реальные повторяющиеся сценарии;
- станет понятно, какие части процесса действительно стоит автоматизировать.

### 18.5. Ручная работа создаёт основу для будущего фреймворка

Сегодняшние решения, исключения и production log могут в будущем стать:

- контрактами характеристик;
- тестовыми примерами;
- правилами нормализации;
- fixtures;
- наборами конфликтов;
- критериями безопасного apply.
