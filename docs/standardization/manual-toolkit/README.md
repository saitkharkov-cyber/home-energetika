# Manual Standardization Toolkit

Дата создания: 16-07-2026

Статус: `active`

## 1. Назначение

Этот каталог содержит лёгкий набор переиспользуемых шаблонов для ручной стандартизации характеристик OpenCart.

Это не отдельный программный продукт, не автоматический pipeline и не продолжение `framework-standardization`.

Рабочий процесс остаётся ручным:

`SQL-инвентаризация → семантический анализ → решение человека → один изменяющий SQL → проверка → production-log`

Toolkit нужен только для того, чтобы:

- не писать диагностические запросы заново;
- не забывать обязательные проверки;
- сохранять одинаковый порядок работы;
- быстрее начинать следующую категорию или характеристику;
- не смешивать временные выгрузки с постоянной документацией.

## 2. Базовые ограничения

Для каждой операции обязательно:

- использовать только прямой охват категории через `oc_product_to_category`;
- явно указывать `category_id`;
- явно указывать `language_id`;
- работать только с одной смысловой группой характеристик за раз;
- сначала выполнять `SELECT`;
- перед изменением получать точное ожидаемое количество строк;
- выполнять один изменяющий SQL за шаг;
- после изменения выполнять отдельный post-check;
- не изменять данные вне утверждённой области;
- не удалять глобальные определения атрибутов без отдельного решения;
- не считать предложение ChatGPT разрешением на изменение данных;
- неоднозначные случаи оставлять на ручную проверку.

## 3. Прямой охват категории

Все category-scoped запросы должны использовать:

```sql
INNER JOIN oc_product_to_category AS p2c
    ON p2c.product_id = pa.product_id
   AND p2c.category_id = {{CATEGORY_ID}}
```

Подкатегории автоматически не включаются.

Запросы по характеристикам должны дополнительно ограничиваться:

```sql
pa.language_id = {{LANGUAGE_ID}}
```

Стандартное значение для текущего каталога:

```text
LANGUAGE_ID = 1
```

## 4. Рабочий цикл

### Шаг 1. Зафиксировать область

Перед началом записать:

```text
category_id:
название категории:
language_id:
смысловая группа:
дата:
```

### Шаг 2. Выполнить инвентаризацию категории

Использовать:

```text
sql/01_category_inventory.sql
```

Результат должен показать:

- все атрибуты прямого охвата;
- название атрибута;
- группу атрибута;
- количество строк;
- количество уникальных товаров;
- общее количество товаров в прямом охвате;
- процент охвата категории.

### Шаг 3. Выбрать одну характеристику или смысловую группу

Не объединять в одном решении соседние, но разные характеристики.

Примеры:

- максимальный, минимальный и номинальный напор — разные характеристики;
- максимальная, минимальная и номинальная производительность — разные характеристики;
- диаметр насоса, подключения, трубы и скважины — разные характеристики;
- мощность двигателя и потребляемая мощность — разные характеристики.

### Шаг 4. Получить распределение значений

Использовать:

```text
sql/02_attribute_distribution.sql
```

Минимальные параметры:

```text
{{CATEGORY_ID}}
{{LANGUAGE_ID}}
{{ATTRIBUTE_ID}}
```

Проверить:

- все варианты значений;
- количество строк;
- количество товаров;
- производителей;
- репрезентативные примеры товаров;
- последовательные локальные импортные блоки;
- подозрительные единичные значения.

### Шаг 5. Проверить качество данных

Использовать:

```text
sql/03_attribute_quality.sql
```

Проверить:

- пустые значения;
- дубли одного атрибута у товара;
- несколько разных значений одного атрибута у товара;
- пробелы по краям;
- единицы измерения внутри значения;
- десятичные запятые;
- текст в числовой характеристике;
- диапазоны;
- приблизительные и составные значения.

### Шаг 6. Проверить канон и aliases

Использовать:

```text
sql/04_canonical_alias_overlap.sql
```

Ожидаемые группы:

```text
canonical_only
alias_only
both_same
both_conflict
```

До переноса данных обязательно понять:

- какие строки можно перенести безопасно;
- где канон уже заполнен;
- где значения совпадают;
- где присутствует настоящий конфликт;
- какие attributes нельзя объединять по смыслу.

### Шаг 7. Принять смысловое решение

Решение принимает человек после анализа данных и обсуждения с ChatGPT.

Нужно зафиксировать:

- канонический атрибут;
- каноническое название;
- единицу измерения;
- тип значения;
- aliases;
- допустимые преобразования;
- запрещённые соседние характеристики;
- исключения;
- строки для ручной проверки;
- ожидаемый итог.

Для сложной характеристики использовать:

```text
templates/ATTRIBUTE_WORKSHEET.md
```

### Шаг 8. Выполнить preview

Использовать:

```text
sql/05_change_preview.sql
```

Preview должен содержать минимум:

- `product_id`;
- название товара;
- исходный `attribute_id`;
- целевой `attribute_id`;
- исходное значение;
- предполагаемое новое значение;
- существующее значение целевого атрибута, если оно есть;
- статус preview: `target_missing`, `target_same` или `target_conflict`.

Количество строк preview должно точно совпадать с ожидаемым количеством строк изменяющего SQL.

### Шаг 9. Выполнить один изменяющий SQL

Изменяющий SQL выполняется только после подтверждения решения.

Не объединять в одном запросе разные смысловые операции без необходимости.

После выполнения зафиксировать:

```text
ожидалось строк:
изменено строк:
время выполнения:
```

### Шаг 10. Выполнить post-check

Использовать:

```text
sql/06_postcheck.sql
```

Проверить:

- итоговое количество строк канона;
- количество уникальных товаров;
- остаток aliases;
- дубли;
- пустые значения;
- неканонические значения;
- товары одновременно с каноном и alias;
- соответствие фактического результата ожидаемому.

Для проверки недопустимых значений используется маркер:

```text
{{NONCANONICAL_CONDITION}}
```

Пример:

```sql
TRIM(CAST(pa.text AS CHAR)) NOT IN ('220', '380')
```

### Шаг 11. Зафиксировать результат

Использовать:

```text
templates/PRODUCTION_LOG_ENTRY.md
```

Готовую запись добавить в:

```text
docs/standardization/PRODUCTION_STANDARDIZATION_LOG.md
```

Production-log является накопительным журналом подтверждённых решений.

### Шаг 12. Выполнить финальный аудит категории

После завершения всех характеристик использовать:

```text
sql/07_final_category_audit.sql
```

Аудит должен проверять:

```text
UNEXPECTED_ATTRIBUTE
MISSING_ATTRIBUTE
NAME_MISMATCH
EXPECTED_COUNT_NOT_SET
ROW_COUNT_MISMATCH
DUPLICATES
EMPTY_VALUES
NONCANONICAL_VALUES
OK
```

Категория считается завершённой только после проверки всех утверждённых канонов и получения итогового статуса `OK` по каждой ожидаемой характеристике.

## 5. Параметры SQL-шаблонов

В шаблонах используются текстовые маркеры:

```text
{{CATEGORY_ID}}
{{LANGUAGE_ID}}
{{ATTRIBUTE_ID}}
{{CANONICAL_ATTRIBUTE_ID}}
{{ALIAS_ATTRIBUTE_IDS}}
{{SOURCE_ATTRIBUTE_ID}}
{{TARGET_ATTRIBUTE_ID}}
{{NEW_VALUE_EXPRESSION}}
{{SOURCE_FILTER_CONDITION}}
{{NONCANONICAL_CONDITION}}
```

Перед выполнением запроса каждый маркер должен быть заменён фактическим значением.

Не выполнять SQL с незаменёнными маркерами.

## 6. Реестр канонов

Фактический production-реестр:

```text
oc_canonical_attributes
```

Читаемый документальный снимок:

```text
CANONICAL_REGISTRY.md
```

При расхождении источником фактического состояния является production-база.

В реестре должны храниться только подтверждённые каноны.

Минимальные поля:

```text
canonical_code
attribute_id
название
группа
тип значения
единица
допустимые значения или формат
status
locked
примечание
```

## 7. Что хранить в Git

Разрешено хранить:

- универсальные SQL-шаблоны;
- инструкции;
- подтверждённые канонические решения;
- production-log;
- шаблоны worksheet и post-check;
- финальные критерии аудита.

## 8. Что не хранить в Git

Не добавлять:

- SQL dump production-базы;
- выгрузки phpMyAdmin;
- CSV со списками товаров;
- временные результаты запросов;
- реквизиты подключения;
- пароли;
- одноразовые backup-файлы;
- локальные snapshots;
- готовые массовые `UPDATE` и `DELETE`, если они не нужны как документированный пример;
- полные описания товаров, выгруженные из production.

## 9. Методические правила

### 9.1. Техническая корректность не равна полезности

Атрибут не нужно сохранять только потому, что его значение технически корректно.

Нужно учитывать:

- помогает ли он покупателю сравнивать товары;
- заполнен ли он системно;
- не создаёт ли неполное заполнение ложного впечатления;
- не зависит ли значение от комплектации;
- не повторяет ли характеристика смысл категории.

### 9.2. Малый охват не является автоматическим основанием для удаления

Атрибут нельзя удалять только из-за небольшого процента охвата.

Нужно отдельно проверить:

- источник блока;
- смысл;
- пользу;
- возможность системного заполнения;
- риск ложного отсутствия;
- наличие подтверждения в описаниях или документации.

### 9.3. Единица находится в названии

Для числовых характеристик:

```text
название: Максимальный напор (м)
значение: 75
```

Нежелательно:

```text
название: Максимальный напор
значение: 75 м
```

### 9.4. Семантические решения не выводятся из чисел

Нельзя автоматически считать одинаковые числа подтверждением одинакового смысла.

Примеры:

- `125` в модели товара не подтверждает номинальную производительность;
- `90` в описании упаковки не подтверждает диаметр скважины;
- `230/400` как паспортные обозначения не обязательно являются отдельными канонами каталога.

## 10. Текущий эталон

Категория:

```text
11900213 — Скважинные насосы
```

Итог:

```text
18 канонических атрибутов
18 / 18 OK
дубли: 0
пустые значения: 0
неканонические значения: 0
неожиданные атрибуты: 0
```

Эта категория используется как эталон для проверки SQL-шаблонов toolkit.

## 11. Граница проекта

Manual Standardization Toolkit:

- не подключается к production самостоятельно;
- не выполняет SQL;
- не принимает решения;
- не вызывает OpenAI API;
- не заменяет ручную проверку;
- не является частью `framework-standardization`;
- не меняет `framework-standardization/docs/HANDOFF.md`.

Он только стандартизирует ручной рабочий процесс.
