## Raw values inventory

**Raw values inventory** — это список реальных сырых значений характеристики, которые уже есть в базе.

Raw values — это значения “как они записаны сейчас”, до нормализации.

Inventory — это инвентаризация, то есть аккуратный список: какие значения есть, сколько раз встречаются, у каких `attribute_id`, в каких товарах и какие есть проблемы.

### Простыми словами

Raw values inventory отвечает на вопрос:

**Что реально лежит в базе по выбранной группе характеристик?**

Например, для характеристики `Максимальный напор` в базе могут быть такие raw values:

- `46.5м.`;
- `68м.`;
- `93м.`;
- `133м.`;
- `20,5 м`;
- `100 м`;
- пустое значение;
- значение с ошибкой;
- значение в другом формате.

На этом этапе мы ещё не исправляем значения.

Мы только собираем и показываем картину.

### Почему inventory идёт после canonical selection

Raw values inventory нельзя делать слишком рано.

Сначала нужно понять, какие `attribute_id` входят в выбранную группу:

- какой `attribute_id` является canonical;
- какие `attribute_id` пользователь подтвердил как aliases / дубли;
- какие похожие `attribute_id` пользователь явно исключил;
- какие кандидаты остались unresolved.

Если сначала не выбрать группу, можно случайно смешать разные смыслы.

Например:

- `Максимальный напор`;
- `Минимальный напор`;
- `Номинальный напор`;
- `Max напор, м`.

Названия похожи, но не все они обязательно означают одно и то же.

### Место в workflow

Raw values inventory находится после canonical selection и перед unit contract:

`target attribute meaning -> discovery -> canonical selection -> include/exclude aliases -> raw values inventory -> canonical unit / normalized_value contract -> proposals -> review-chain -> apply-plan`

То есть сначала пользователь выбирает, какие `attribute_id` относятся к одной характеристике.

Потом система собирает реальные значения по этой утверждённой группе.

### Что показывает inventory

Inventory может показывать:

- `raw_value`;
- сколько раз значение встречается;
- в каких `attribute_id` оно встречается;
- с какими `attribute_name` связано;
- примеры `product_id`;
- примеры названий товаров, если доступны readonly;
- warnings;
- признаки грязных или неоднозначных данных.

Пример:

- raw_value: `68м.`;
- usage_count: `15`;
- attribute_id: `12`;
- attribute_name: `Максимальный напор`;
- sample_product_ids: `123`, `456`, `789`;
- warning: нет.

Другой пример:

- raw_value: `20,5 м`;
- usage_count: `3`;
- warning: comma decimal, нужно правило для запятой.

Ещё пример:

- raw_value: `68`;
- usage_count: `5`;
- warning: unit missing, непонятно, метры это или что-то другое.

### Что inventory не делает

Raw values inventory не выбирает canonical attribute.

Он не решает:

`этот attribute_id главный`

Это уже должно быть решено раньше.

Raw values inventory не объединяет похожие характеристики автоматически.

Он не решает:

`Максимальный напор` и `Номинальный напор` — это одно и то же`

Это решает человек на этапе canonical selection.

Raw values inventory не утверждает единицу измерения.

Он не решает:

`всё это точно метры`

Он только показывает, что в raw values часто встречается `м`, `м.`, `м вод. ст.` или другие варианты.

Raw values inventory не создаёт final `normalized_value`.

Он не решает:

`68м.` -> `68`

Это будет делать следующий этап после утверждённого unit contract.

Raw values inventory не создаёт proposals и не запускает SQL apply.

### Зачем нужны warnings

Inventory должен подсвечивать подозрительные места.

Например:

- пустые значения;
- разные написания одного значения;
- разные единицы измерения;
- число без единицы;
- диапазоны;
- дроби;
- HTML entities;
- запятая вместо точки;
- текст вместе с числом;
- слишком много уникальных значений;
- значения, похожие на другую характеристику;
- значения из alias-атрибута, которые конфликтуют с canonical meaning.

Warnings нужны, чтобы человек понял:

`эти данные можно нормализовать автоматически`

или

`здесь нужно отдельное правило / ручной review`

### Пример из жизни

Представь склад.

Перед тем как наводить порядок, нужно сначала сделать инвентаризацию.

Нельзя сразу переклеивать все ярлыки на коробках.

Сначала нужно понять:

- какие коробки вообще есть;
- сколько их;
- какие подписаны одинаково;
- где подписи разные;
- где подписи странные;
- где коробка похожа на нужную, но внутри может быть другое.

Raw values inventory — это такая инвентаризация, только для значений характеристик.

### Пример в нашем проекте

Для `pump_max_head` inventory мог бы показать:

- `46.5м.` встречается несколько раз;
- `68м.` встречается несколько раз;
- `93м.` встречается несколько раз;
- `20,5 м` требует правила для запятой;
- пустые значения требуют warning;
- значения без `м` требуют manual review;
- значения с другой единицей требуют отдельного решения.

После этого пользователь может утвердить contract:

- canonical unit: `m`;
- normalized_value: decimal number in meters;
- comma decimal разрешён или запрещён;
- значения без единицы отправлять в manual review или обрабатывать по правилу.

### Главная мысль

**Raw values inventory — это не исправление данных.**

Это честный список того, что реально есть в базе.

Он нужен, чтобы перед нормализацией не гадать, а видеть факты.

Правило проекта:

`raw values inventory должен быть completed перед canonical unit / normalized_value contract`

И ещё одно правило:

`нельзя генерировать proposals, пока inventory не завершён и unit contract не утверждён`
