## Основная идея проекта

Проект строит framework для безопасной стандартизации характеристик каталога.

Ключевая задача:

```text
найти разные варианты одной характеристики
и привести их к утверждённому каноническому атрибуту OpenCart
```

Например:

```text
Диаметр насоса (мм)
Диаметр корпуса
Диаметр, мм
Ø насоса
```

могут быть рассмотрены как кандидаты на приведение к одному каноническому атрибуту.

---

## Главный архитектурный принцип

Канонический атрибут является глобальной сущностью OpenCart.

Категория не является частью идентичности канона.

Категория используется только как область анализа и применения.

```text
canonical attribute
→ глобальный целевой атрибут OpenCart

category / scope
→ где сейчас выполняется анализ
```

---

## Что хранится в БД

На текущем этапе в БД выносится только таблица:

```text
{DB_PREFIX}canonical_attributes
```

Она отвечает только на один вопрос:

```text
какой реальный attribute_id OpenCart считается глобальным каноном
```

В БД не выносятся:

* scope / category_id;
* value_parser;
* value_type;
* allow_empty;
* правила нормализации;
* правила валидации;
* синонимы;
* результаты анализа.

Эти данные остаются внутри Framework, `Attribute Job` или конфигурации запуска.

## Текущий статус

На текущем этапе зафиксирован архитектурный контракт конвейера обработки атрибутов.

Приняты ключевые решения:

* Framework работает с одной задачей обработки характеристики (`Attribute Job`);
* канон является глобальным атрибутом OpenCart;
* категория используется как scope анализа;
* в БД хранится только реестр канонических атрибутов;
* `AttributeExporter` только читает факты из БД;
* `AnalyzeNamesStage` анализирует имена и кандидатов;
* `AnalyzeValuesStage` анализирует значения через `ValueParser`;
* `ValueParser` нормализует отдельные значения;
* Framework не изменяет БД и не применяет SQL автоматически;
* существующие импорты пока не изменяются;
* будущий `CanonicalAttributeResolver` рассматривается как точка интеграции с импортами.
