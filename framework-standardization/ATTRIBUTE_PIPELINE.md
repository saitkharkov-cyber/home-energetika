# Конвейер обработки атрибутов

## Назначение

Конвейер обработки атрибутов определяет единый жизненный цикл обработки любой характеристики каталога.

Конвейер не зависит от конкретной характеристики.

Он одинаково работает для:

- максимального напора;
- производительности;
- диаметра;
- напряжения;
- любой новой характеристики, добавленной в будущем.

Конкретная характеристика определяется исключительно конфигурацией.

---

# Основной принцип

Единицей обработки Фреймворка является **задача обработки характеристики** (`Attribute Job`).

Она представляет собой задачу стандартизации одной характеристики одной категории товаров.

Например:

```text
Диаметр насоса
Категория: Скважинные насосы
```

или

```text
Максимальный напор
Категория: Скважинные насосы
```

Фреймворк ничего не знает о смысле характеристики.

Он лишь выполняет конвейер обработки для указанной задачи.

---

# Конвейер обработки

```text
Задача обработки характеристики
            ↓
Загрузка
            ↓
Анализ
            ↓
Нормализация
            ↓
Проверка
            ↓
Генерация SQL
            ↓
Формирование отчёта
            ↓
Неизвестные значения
```

Каждый этап имеет собственную ответственность.

---

# Движок конвейера

Фреймворк реализуется как **движок конвейера** (`Pipeline Engine`).

Движок ничего не знает о конкретной характеристике, парсерах, SQL или отчётах.

Его единственная задача — выполнить заданную последовательность этапов над одним **контекстом характеристики** (`AttributeContext`).

Базовый контракт движка:

```text
Контекст характеристики
            ↓
Этап 1
            ↓
Контекст характеристики
            ↓
Этап 2
            ↓
Контекст характеристики
            ↓
...
```

Условно работу движка можно представить так:

```python
for stage in pipeline:
    context = stage.run(context)
```

Конвейер не является частью бизнес-логики.

Конвейер — это последовательность этапов, которую выполняет движок.

---

# Модель этапов

Конвейер состоит из независимых этапов.

Каждый этап:

- имеет единственную ответственность;
- принимает `AttributeContext`;
- возвращает `AttributeContext`;
- не управляет другими этапами;
- не знает устройство всего конвейера;
- не изменяет БД напрямую;
- не применяет SQL самостоятельно.

Примеры этапов:

- `LoadStage`
- `AnalyzeStage`
- `NormalizeStage`
- `ValidateStage`
- `GenerateSqlStage`
- `GenerateReportStage`
- `UnknownValuesStage`

---

# Контекст характеристики

**Контекст характеристики** (`AttributeContext`) — единый объект состояния, который проходит через весь конвейер.

Этапы не передают друг другу разрозненные параметры.

Все промежуточные результаты помещаются в `AttributeContext`.

По мере прохождения конвейера он постепенно обогащается данными.

Например:

```text
Задача обработки характеристики
            ↓
Этап загрузки
            ↓
товары
исходные значения

            ↓
Этап анализа
            ↓
уникальные значения
статистика

            ↓
Этап нормализации
            ↓
нормализованные значения

            ↓
Этап проверки
            ↓
статусы

            ↓
Этап генерации SQL
            ↓
SQL

            ↓
Этап формирования отчёта
            ↓
отчёт

            ↓
Этап неизвестных значений
            ↓
неизвестные значения
```

---

# Независимость этапов

Этапы не взаимодействуют друг с другом напрямую.

Единственным способом обмена данными между этапами является `AttributeContext`.

Это означает:

- этап не вызывает другие этапы;
- этап не знает, какой этап был до него;
- этап не знает, какой этап будет после него;
- этап не использует внутренние структуры других этапов.

Каждый этап работает исключительно с текущим состоянием `AttributeContext`.

Благодаря этому:

- этапы можно менять местами (если это допускает логика конвейера);
- этап можно заменить альтернативной реализацией;
- новые этапы можно добавлять без изменения существующих;
- конвейер остаётся расширяемым без изменения движка.

---

# Этап 1. Загрузка

### Ответственность

Получить исходные данные.

### Вход

- category_id;
- attribute_id.

### Выход

Список исходных значений.

---

# Этап 2. Анализ

### Ответственность

Проанализировать полученные данные.

### Результат

- список уникальных значений;
- статистика повторяемости;
- предварительная диагностика.

Этап не изменяет данные.

---

# Этап 3. Нормализация

### Ответственность

Вызвать соответствующий парсер (`Parser`).

Парсер отвечает исключительно за преобразование одного значения.

Парсер:

- не работает с БД;
- не знает о SQL;
- не знает о других характеристиках;
- не выполняет проверку результатов.

---

# Этап 4. Проверка

### Ответственность

Проверить результат нормализации.

### Возможные статусы

- `OK`;
- `UNKNOWN`;
- `AMBIGUOUS`.

Фреймворк не пытается автоматически исправлять `UNKNOWN`.

---

# Этап 5. Генерация SQL

### Ответственность

Создать SQL только для записей со статусом `OK`.

Фреймворк никогда самостоятельно не применяет SQL.

---

# Этап 6. Формирование отчёта

### Ответственность

Создать отчёт обработки.

Например:

- количество товаров;
- количество уникальных значений;
- количество успешно обработанных;
- количество неизвестных значений.

---

# Этап 7. Неизвестные значения

### Ответственность

Сформировать список значений, требующих ручной обработки.

Фреймворк не пытается автоматически интерпретировать неизвестные значения.

---

# Инварианты конвейера

Конвейер никогда:

- не изменяет БД;
- не применяет SQL;
- не исправляет `UNKNOWN`;
- не зависит от конкретной характеристики.

Конвейер всегда:

- использует парсер, определённый конфигурацией;
- выполняет этапы в порядке, заданном конвейером;
- формирует SQL;
- формирует отчёт;
- формирует список неизвестных значений.

---

# Расширение Фреймворка

Добавление новой характеристики не должно требовать изменения конвейера.

Конвейер архитектурно стабилен.

Расширение выполняется путём добавления:

- новой конфигурации;
- нового парсера (при необходимости).

Новые возможности Фреймворка должны добавляться через новые этапы, а не через изменение движка конвейера.

Например:

- `ExcelExportStage`;
- `HtmlReportStage`;
- `QualityControlStage`;
- `StatisticsStage`.

Ядро Фреймворка при этом остаётся неизменным.

Расширение достигается добавлением новых этапов, а не изменением существующего конвейера.

---

# Итоговая схема

```text
Задача обработки характеристики
            ↓
Движок конвейера
            ↓
Создание контекста характеристики
            ↓
Этап загрузки
            ↓
Этап анализа
            ↓
Этап нормализации
            ↓
Этап проверки
            ↓
Этап генерации SQL
            ↓
Этап формирования отчёта
            ↓
Этап неизвестных значений
            ↓
Результат обработки
```

Эта схема подводит итог архитектуре.

Фреймворк решает задачу обработки характеристики посредством движка конвейера, который создаёт единый контекст характеристики и последовательно передаёт его через независимые этапы обработки до получения итогового результата.
