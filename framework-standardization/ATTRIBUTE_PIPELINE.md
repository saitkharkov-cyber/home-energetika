# Конвейер обработки атрибутов

## Назначение

Конвейер обработки атрибутов определяет единый жизненный цикл обработки любой характеристики каталога.

Конвейер не зависит от конкретной характеристики.

Он одинаково работает для:

* максимального напора;
* производительности;
* диаметра;
* напряжения;
* любой новой характеристики, добавленной в будущем.

Конкретная характеристика определяется исключительно конфигурацией и входной задачей обработки.

---

# Основной принцип

Единицей обработки Фреймворка является **задача обработки характеристики** (`Attribute Job`).

Она представляет собой задачу стандартизации одной выбранной характеристики в заданной области каталога.

Например:

```text
Диаметр насоса
Категория: Скважинные насосы
```

или

```text
Максимальный напор
Категория: Скважинные насосы
```

Фреймворк ничего не знает о бизнес-смысле характеристики.

Он лишь выполняет конвейер обработки для указанной задачи.

---

# Канонические характеристики

Перед стандартизацией характеристики необходимо определить, существует ли уже подходящая **глобальная каноническая характеристика**.

Каноническая характеристика — это утверждённый атрибут OpenCart, который считается целевым стандартом для данной характеристики по смыслу.

Например:

```text
Напряжение, В
```

может использоваться как каноническая характеристика для разных категорий:

* скважинные насосы;
* фекальные насосы;
* насосные станции;
* другие категории, где эта характеристика имеет тот же смысл.

---

# Глобальный канон и область применения

Каноническая характеристика является **глобальной сущностью**, потому что в OpenCart атрибуты не привязаны к категории как отдельные сущности.

Импорт также работает не по категории, а по:

* `attribute_id`;
* имени атрибута;
* группе атрибута;
* заранее настроенному соответствию.

Категория используется не как часть идентичности канона, а как **область безопасного анализа и применения**.

Правило:

```text
Канонический атрибут — глобальная сущность OpenCart.
Категория — область анализа, поиска синонимов и применения стандартизации.
```

Поэтому перед обработкой характеристики в категории сначала проверяется:

```text
Есть ли уже глобальный канонический атрибут, подходящий по смыслу?
```

Если такой атрибут есть, он используется как целевой канон.

Если такого атрибута нет, необходимо вручную создать или утвердить новый глобальный канон.

---

# Реестр канонических характеристик

Для работы со стандартизацией нужен отдельный реестр канонических характеристик.

Рабочее название:

```text
canonical_attributes
```

Назначение таблицы:

```text
Хранить список утверждённых глобальных канонических характеристик,
которые могут использоваться AttributeExporter, Framework и будущим импортом.
```

Таблица канонов не хранит синонимы.

Она хранит только утверждённые целевые характеристики.

Минимальная структура:

```text
canonical_attributes
├─ id
├─ target_oc_attribute_id
├─ canonical_attribute_name
├─ canonical_attribute_group_id
├─ canonical_attribute_group_name
├─ status
├─ locked
├─ comment
├─ created_at
└─ updated_at
```

Где:

* `id` — `canonical_id`, ID записи в нашем реестре канонов;
* `target_oc_attribute_id` — реальный `attribute_id` в OpenCart, используемый как целевой канон;
* `canonical_attribute_name` — контрольное имя канонического атрибута;
* `canonical_attribute_group_id` — группа атрибута в OpenCart;
* `canonical_attribute_group_name` — контрольное имя группы;
* `status` — состояние канона;
* `locked` — признак защиты от случайных изменений;
* `comment` — инженерный комментарий.

Правила обработки значений не являются частью идентичности глобального канона.

`value_parser` относится к этапу нормализации.

`value_type` и `allow_empty` относятся к этапу проверки или к конфигурации обработки конкретной задачи.

---

# Области применения канона

Так как канон глобальный, но стандартизация выполняется в рамках конкретных категорий, область применения должна храниться отдельно.

Рабочее название:

```text
canonical_attribute_scopes
```

Назначение:

```text
Определить, в каких категориях разрешено использовать конкретный канонический атрибут.
```

Минимальная структура:

```text
canonical_attribute_scopes
├─ id
├─ canonical_id
├─ category_id
├─ status
├─ comment
├─ created_at
└─ updated_at
```

Где:

* `canonical_id` — ссылка на запись в `canonical_attributes`;
* `category_id` — категория, в которой этот канон разрешён к применению;
* `status` — активна ли область применения;
* `comment` — пояснение или примечание.

Таким образом:

```text
canonical_attributes
→ что является каноном по смыслу

canonical_attribute_scopes
→ где этот канон разрешено применять
```

Scope обязателен всегда как область анализа и применения.

Физическая таблица `canonical_attribute_scopes` может быть отложена только если scope временно приходит из `Attribute Job` или конфигурации запуска.

---

# Синонимы и разновидности характеристики

В БД может существовать несколько разновидностей одной и той же характеристики.

Например:

```text
Диаметр насоса (мм)
Диаметр корпуса
Диаметр, мм
Ø насоса
```

Задача стандартизации — найти такие разновидности в заданной области и привести их к утверждённому каноническому атрибуту.

Синонимы не являются частью реестра канонов.

Синонимы выявляются в процессе анализа конкретной категории.

На текущем этапе синонимы не применяются автоматически к продуктиву.

Фреймворк только готовит результат для ручной проверки.

---

# Роль AttributeExporter

`AttributeExporter` — это read-only слой чтения фактического состояния БД.

Он нужен для выгрузки атрибутов, значений и статистики из OpenCart в рамках заданной области анализа.

`AttributeExporter` не является парсером значений.

Он не нормализует отдельные значения характеристики.

Он не ищет и не утверждает синонимы семантически.

Он получает задачу обработки и собирает данные из БД.

Вход `AttributeExporter`:

```text
Attribute Job
├─ canonical_attribute
└─ category_scope
```

`AttributeExporter` должен:

* проверить наличие целевого OpenCart-атрибута в системе;
* получить данные по заданной категории;
* вернуть найденные сырые атрибуты;
* посчитать `usage_count` каждого найденного атрибута;
* собрать `sample_values`;
* вернуть результат во Фреймворк.

`AttributeExporter` не должен:

* изменять БД;
* создавать атрибуты;
* удалять атрибуты;
* нормализовать значения;
* использовать `value_parser`;
* использовать `value_type`;
* использовать `allow_empty`;
* оценивать найденные атрибуты как синонимы;
* принимать финальное решение о синонимах;
* применять результат к продуктиву.

---

# Результат AttributeExporter

`AttributeExporter` возвращает `AttributeExportResult`.

Минимальная структура:

```text
AttributeExportResult
├─ target_attribute_check
│  ├─ status
│  ├─ target_oc_attribute_id
│  ├─ target_oc_attribute_name
│  └─ warnings[]
│
├─ canonical_attribute
│  ├─ canonical_id
│  ├─ target_oc_attribute_id
│  ├─ attribute_name
│  ├─ attribute_group_id
│  └─ attribute_group_name
│
├─ category_scope
│  └─ category_id
│
├─ found_attributes[]
│  ├─ attribute_id
│  ├─ attribute_name
│  ├─ attribute_group_id
│  ├─ attribute_group_name
│  ├─ usage_count
│  ├─ values_count
│  ├─ unique_values_count
│  └─ sample_values[]
│
└─ warnings[]
```

Частотность является обязательной частью результата `AttributeExporter`.

На этом уровне частотность является только фактом из БД.

`AttributeExporter` не делает вывод, является ли найденный атрибут синонимом, дублем или главным кандидатом.

---

# Attribute Job

`Attribute Job` — входная задача обработки одной выбранной характеристики.

Минимальная структура:

```text
Attribute Job
├─ canonical_attribute
│  ├─ canonical_id
│  ├─ target_oc_attribute_id
│  ├─ attribute_name
│  ├─ attribute_group_id
│  └─ attribute_group_name
│
├─ category_scope
│  └─ category_id
│
├─ value_parser
├─ value_type
├─ allow_empty
├─ output_format
└─ metadata
```

Смысл:

```text
Attribute Job = что обработать, в какой области, по каким правилам и какой результат подготовить.
```

В `Attribute Job` не помещаются промежуточные данные этапов.

Промежуточное состояние живёт в `AttributeContext`.

---

# Контекст характеристики

**Контекст характеристики** (`AttributeContext`) — единый объект состояния, который проходит через весь конвейер.

`AttributeContext` создаётся Фреймворком.

`AttributeExporter` или внешний Runner не должны создавать внутреннее состояние Фреймворка.

Этапы не передают друг другу разрозненные параметры.

Все промежуточные результаты помещаются в `AttributeContext`.

Минимально `AttributeContext` содержит:

```text
AttributeContext
├─ attribute_job
├─ attribute_export_result
├─ attribute_name_structure
├─ attribute_value_structure
├─ analysis_result
├─ normalized_values
├─ validation_result
├─ sql_preview
├─ report
└─ unknown_values
```

Отдельно фиксируется разделение:

```text
attribute_name_structure
→ разобранная структура названия характеристики

attribute_value_structure
→ разобранная структура значения характеристики
```

Название характеристики и значение характеристики не смешиваются.

Это разные модели, разные проверки и разные ошибки.

---

# Конвейер обработки

```text
Задача обработки характеристики
            ↓
Загрузка
            ↓
Анализ
            ↓
Нормализация
            ↓
Проверка
            ↓
Генерация SQL
            ↓
Формирование отчёта
            ↓
Неизвестные значения
```

Каждый этап имеет собственную ответственность.

---

# Движок конвейера

Фреймворк реализуется как **движок конвейера** (`Pipeline Engine`).

Движок ничего не знает о конкретной характеристике, экспортёрах, парсерах значений, SQL или отчётах.

Его единственная задача — выполнить заданную последовательность этапов над одним **контекстом характеристики** (`AttributeContext`).

Базовый контракт движка:

```text
Контекст характеристики
            ↓
Этап 1
            ↓
Контекст характеристики
            ↓
Этап 2
            ↓
Контекст характеристики
            ↓
...
```

Условно работу движка можно представить так:

```python
for stage in pipeline:
    context = stage.run(context)
```

Конвейер не является частью бизнес-логики.

Конвейер — это последовательность этапов, которую выполняет движок.

---

# Модель этапов

Конвейер состоит из независимых этапов.

Каждый этап:

* имеет единственную ответственность;
* принимает `AttributeContext`;
* возвращает `AttributeContext`;
* не управляет другими этапами;
* не знает устройство всего конвейера;
* не изменяет БД напрямую;
* не применяет SQL самостоятельно.

Примеры этапов:

* `LoadStage`
* `AnalyzeStage`
* `NormalizeStage`
* `ValidateStage`
* `GenerateSqlStage`
* `GenerateReportStage`
* `UnknownValuesStage`

---

# Независимость этапов

Этапы не взаимодействуют друг с другом напрямую.

Единственным способом обмена данными между этапами является `AttributeContext`.

Это означает:

* этап не вызывает другие этапы;
* этап не знает, какой этап был до него;
* этап не знает, какой этап будет после него;
* этап не использует внутренние структуры других этапов.

Каждый этап работает исключительно с текущим состоянием `AttributeContext`.

Благодаря этому:

* этапы можно менять местами, если это допускает логику конвейера;
* этап можно заменить альтернативной реализацией;
* новые этапы можно добавлять без изменения существующих;
* конвейер остаётся расширяемым без изменения движка.

---

# Этап 1. Загрузка

### Ответственность

Получить исходные данные для задачи обработки.

На этом этапе `AttributeExporter` читает фактическое состояние БД в рамках заданной области.

### Вход

* `Attribute Job`;
* канонический атрибут;
* область анализа;
* категория.

### Выход

* `AttributeExportResult`;
* найденные сырые атрибуты;
* частотность использования;
* примеры значений;
* предупреждения.

---

# Этап 2. Анализ

### Ответственность

Проанализировать полученные данные.

### Результат

* список найденных вариантов характеристики;
* оценка найденных атрибутов как кандидатов;
* статистика повторяемости;
* список уникальных значений;
* предварительная диагностика;
* предупреждения о несоответствиях.

Частотность используется только как диагностический сигнал.

Она не выбирает канон автоматически.

Если выбранный канон не является самым частотным вариантом, это должно быть отражено как предупреждение, а не как автоматическая ошибка.

Этап не изменяет данные.

---

# Этап 3. Нормализация

### Ответственность

Вызвать соответствующий парсер значений (`ValueParser`).

`value_parser` относится к `NormalizeStage`.

Парсер значений отвечает исключительно за преобразование одного значения.

Парсер значений:

* не работает с БД;
* не знает о SQL;
* не знает о других характеристиках;
* не выполняет запись результатов;
* не принимает решение о публикации.

---

# Этап 4. Проверка

### Ответственность

Проверить результат нормализации.

`value_type` и `allow_empty` относятся к `ValidateStage` или к конфигурации обработки конкретной задачи.

### Возможные статусы

* `OK`;
* `UNKNOWN`;
* `AMBIGUOUS`.

Фреймворк не пытается автоматически исправлять `UNKNOWN`.

Если результат неоднозначен, он должен быть передан на ручную проверку.

---

# Этап 5. Генерация SQL

### Ответственность

Создать SQL только для записей со статусом `OK`.

Фреймворк никогда самостоятельно не применяет SQL.

SQL является preview-результатом для ручной проверки.

---

# Этап 6. Формирование отчёта

### Ответственность

Создать отчёт обработки.

Например:

* количество найденных атрибутов-кандидатов;
* частотность каждого варианта;
* количество уникальных значений;
* количество успешно обработанных значений;
* количество неизвестных значений;
* список предупреждений;
* список потенциальных синонимов;
* список отклонённых кандидатов.

---

# Этап 7. Неизвестные значения

### Ответственность

Сформировать список значений, требующих ручной обработки.

Фреймворк не пытается автоматически интерпретировать неизвестные значения.

---

# Результат Framework

Фреймворк возвращает результат обработки, пригодный для ручной проверки.

`FrameworkResult` — это финальная проекция из `AttributeContext`, а не отдельное независимое состояние.

Минимальная структура:

```text
FrameworkResult
├─ canonical_attribute
├─ proposed_synonym_candidates[]
├─ rejected_candidates[]
├─ value_report
├─ warnings[]
├─ sql_preview
├─ report
└─ unknown_values
```

Фреймворк должен возвращать не только результат, но и объяснение решений.

Это необходимо для ручного контроля стандартизации.

---

# Связь с импортом

Существующие импорты на текущем этапе не изменяются.

Импорт рассматривается как будущий потребитель canonical layer.

Архитектура должна быть спроектирована так, чтобы позже можно было логично подключить защитный слой:

```text
CanonicalAttributeResolver
```

Будущий `CanonicalAttributeResolver` должен решать задачу:

```text
category_id
+ incoming_attribute_name
+ incoming_attribute_group
+ source
→ canonical_id + target_oc_attribute_id / not_found / ambiguous
```

Его будущая роль:

* не дать импорту создать дубль, если канон уже существует;
* не дать записать значение в синоним вместо канона;
* не дать случайно сломать утверждённый канон;
* не дать Suppler создать новый атрибут, если входную характеристику можно сопоставить с каноном;
* дать ExcelPort возможность сопоставлять входные характеристики не только по точному имени.

На текущем этапе импорты не дорабатываются.

Правило:

```text
canonical layer проектируется с учётом будущего импорта,
но сам импорт пока вне зоны реализации.
```

---

# Ручной контроль

Фреймворк не принимает окончательных решений вместо инженера.

Инженер вручную:

* выбирает характеристику для обработки;
* проверяет наличие глобального канона;
* утверждает канонический атрибут;
* задаёт область анализа;
* проверяет найденные синонимы;
* проверяет SQL preview;
* принимает решение о публикации;
* вручную применяет результат на продуктиве.

Фреймворк:

* собирает данные;
* анализирует;
* нормализует;
* валидирует;
* формирует SQL;
* формирует отчёт;
* показывает неизвестные значения;
* не применяет результат автоматически.

---

# Инварианты конвейера

Конвейер никогда:

* не изменяет БД;
* не применяет SQL;
* не исправляет `UNKNOWN`;
* не создаёт атрибуты;
* не удаляет атрибуты;
* не переименовывает атрибуты;
* не публикует результат на продуктив;
* не зависит от конкретной характеристики.

Конвейер всегда:

* работает с одной задачей обработки характеристики;
* использует парсер значений, определённый конфигурацией;
* выполняет этапы в порядке, заданном конвейером;
* формирует SQL preview;
* формирует отчёт;
* формирует список неизвестных значений.

---

# Расширение Фреймворка

Добавление новой характеристики не должно требовать изменения конвейера.

Конвейер архитектурно стабилен.

Расширение выполняется путём добавления:

* новой конфигурации;
* нового парсера значений при необходимости;
* новой записи в реестре канонических характеристик;
* новой области применения канона при необходимости.

Новые возможности Фреймворка должны добавляться через новые этапы, а не через изменение движка конвейера.

Например:

* `ExcelExportStage`;
* `HtmlReportStage`;
* `QualityControlStage`;
* `StatisticsStage`;
* `CanonicalCheckStage`.

Ядро Фреймворка при этом остаётся неизменным.

Расширение достигается добавлением новых этапов, а не изменением существующего конвейера.

# Итоговая схема

```text
Выбор характеристики инженером
            ↓
Проверка глобального канона
            ↓
Определение области анализа
            ↓
Задача обработки характеристики
            ↓
Движок конвейера
            ↓
Создание контекста характеристики
            ↓
Этап загрузки
            ↓
Этап анализа
            ↓
Этап нормализации
            ↓
Этап проверки
            ↓
Этап генерации SQL
            ↓
Этап формирования отчёта
            ↓
Этап неизвестных значений
            ↓
Результат обработки для ручной проверки
```

Эта схема подводит итог архитектуре.

Фреймворк решает задачу обработки характеристики посредством движка конвейера, который создаёт единый контекст характеристики и последовательно передаёт его через независимые этапы обработки до получения итогового результата.

Фреймворк не публикует результат автоматически.

Канонический атрибут является глобальной сущностью OpenCart.

Категория используется как область анализа и применения, но не как часть идентичности канона.
