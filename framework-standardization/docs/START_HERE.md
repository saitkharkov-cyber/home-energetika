# С чего начать — стандартизация характеристик

Дата: 13-07-2026 Europe/Kyiv

## 1. Назначение

Этот документ — постоянная входная инструкция для нового чистого ChatGPT-чата по проекту `HmEnerg_Характеристики / framework-standardization`.

Он не является стартовой инструкцией для Codex.

`START_HERE.md` задаёт:

* роли пользователя, ChatGPT, Codex и документации;
* порядок восстановления контекста новой ChatGPT-сессией;
* lifecycle временного `HANDOFF.md`;
* общие gates и safety-границы;
* правила передачи одного bounded step в Codex.

Основной источник истины — постоянная документация в репозитории и свежие Git-факты. Краткое summary из чата не заменяет документы и не может переопределять свежий Git-state.

## 2. Роли в рабочем процессе

### Пользователь

Пользователь:

* определяет цель и приоритет;
* подтверждает спорные смысловые решения;
* даёт отдельные разрешения на bounded operations;
* управляет gates через отдельный явный `+`;
* определяет protected user-owned changes.

### ChatGPT

ChatGPT является оркестратором работы.

ChatGPT:

* читает постоянную документацию;
* восстанавливает межсессионный контекст;
* получает и проверяет свежие Git-факты;
* выбирает ровно один следующий bounded step;
* не спрашивает Codex: «что делать дальше?»;
* формирует task-specific prompt для Codex;
* явно переносит в prompt все важные scope, gates, запреты и проверки;
* проверяет отчёт, diff и соблюдение scope после работы Codex;
* отдельно управляет documentation updates, commit и push.

### Codex

Codex является исполнителем конкретного ограниченного шага в локальном репозитории.

Codex:

* получает от ChatGPT один task-specific bounded prompt;
* читает только файлы, явно указанные в prompt или непосредственно необходимые для разрешённого шага;
* не определяет самостоятельно текущий target, stage или следующий шаг проекта;
* не получает весь межсессионный контекст вместо конкретной задачи;
* не использует `START_HERE.md` как собственный bootstrap;
* не читает `HANDOFF.md` и `HANDOFF_SPEC.md` в обычной инженерной работе;
* не получает общее поручение «соблюдай `RULES.md`» — значимые ограничения должны быть явно включены ChatGPT в prompt;
* после выполнения разрешённого шага отчитывается и останавливается.

### Документация

Документация хранит устойчивые знания проекта и проверенные evidence.

Она не заменяет решения пользователя и не даёт сама по себе разрешения на implementation, DB, SQL/apply, production/cache, commit или push.

## 3. Порядок входа новой ChatGPT-сессии

Новый ChatGPT должен:

1. подтвердить repository identity, working project и branch;
2. полностью прочитать `framework-standardization/docs/START_HERE.md`;
3. прочитать `framework-standardization/glossary/!_README.md`;
4. прочитать `framework-standardization/docs/DOCUMENTATION_BOUNDARIES.md`;
5. прочитать `framework-standardization/docs/HANDOFF.md`, только если файл существует;
6. получить свежие Git-факты: текущий `HEAD`, `origin/main`, последние commits и `git status --short`;
7. считать свежий Git-state фактом;
8. прочитать `framework-standardization/docs/RULES.md`;
9. прочитать `framework-standardization/docs/DECISIONS.md`;
10. прочитать `framework-standardization/docs/LEGACY_DECISIONS.md`, если задача связана со стандартизацией характеристики, присутствующей в legacy-данных;
11. прочитать `framework-standardization/docs/RUNTIME_CHECKS.md`;
12. прочитать только актуальные specs, необходимые для текущего bounded step.

После восстановления контекста ChatGPT должен вывести пользователю краткий отчёт о Git-state, target, stage, completed/open items, gates, protected changes и одном следующем bounded step.

Никакая реализация не начинается автоматически после чтения документации.

## 4. Lifecycle HANDOFF.md

`framework-standardization/docs/HANDOFF.md` — временный tracked transport artifact между двумя ChatGPT-сессиями.

Его создаёт завершающая ChatGPT-сессия по отдельному пользовательскому разрешению.

Codex не создаёт handoff и не использует его как рабочее задание.

Если `HANDOFF.md` существует, новая ChatGPT-сессия должна:

1. прочитать его после `DOCUMENTATION_BOUNDARIES.md`;
2. получить свежие Git-факты;
3. найти `Session close base commit` в истории текущего `HEAD`;
4. классифицировать изменения после base commit;
5. восстановить target, scope, stage, completed/open items, gates, protected changes и один next bounded step;
6. при blocking inconsistency остановиться и запросить недостающий факт;
7. после успешного восстановления сообщить, что handoff выполнил функцию передачи и готов к lifecycle cleanup.

Удаление `HANDOFF.md`, отдельный cleanup commit и push требуют отдельного пользовательского разрешения.

После cleanup активная работа продолжается без handoff-файла по схеме:

`пользователь ↔ ChatGPT ↔ Codex ↔ документация`.

Если `HANDOFF.md` отсутствует, это штатное состояние активной сессии. ChatGPT продолжает по постоянной документации и текущему диалогу, но не придумывает отсутствующий dynamic target или stage.

`framework-standardization/docs/HANDOFF_SPEC.md` читается ChatGPT только при создании, проверке или cleanup lifecycle handoff. Он не входит в обычный Codex workflow.

## 5. Активная работа после восстановления

После восстановления контекста ChatGPT:

1. выбирает один маленький bounded step;
2. формулирует его пользователю;
3. получает отдельный явный `+` на этот шаг;
4. создаёт точный prompt для Codex;
5. указывает Codex:
   * какие файлы читать;
   * какой scope разрешён;
   * какой один файл или набор строго связанных файлов можно менять;
   * какие действия запрещены;
   * какие проверки выполнить;
   * требуется ли остановка до commit;
6. получает отчёт Codex;
7. проверяет фактический diff и scope;
8. отдельно решает вопрос documentation update, commit и push.

Один prompt Codex не должен незаметно объединять lifecycle cleanup, документационную задачу, инженерный анализ и apply-операцию.

## 6. Где хранится текущее и постоянное состояние

`START_HERE.md` не хранит:

* текущий `HEAD` или stable-point SHA;
* текущий target;
* текущий stage;
* следующий bounded step;
* актуальный backlog;
* результаты последней рабочей сессии.

Эти сведения быстро устаревают.

Источники ответственности:

* свежий Git-state — фактическое состояние репозитория;
* `HANDOFF.md` — временный dynamic snapshot только на границе ChatGPT-сессий;
* текущий диалог — active target и пользовательские разрешения;
* `CURRENT_OVERRIDE.md` — временная существенная корректива внутри активной ChatGPT-сессии;
* `DECISIONS.md` — постоянные архитектурные и процессные решения;
* `RUNTIME_CHECKS.md` — проверенные runtime/manual evidence;
* `LEGACY_DECISIONS.md` — утверждённые legacy-решения;
* инженерные specs — устойчивые границы конкретных компонентов и workflow;
* glossary — значения терминов.

Подробные границы определены в `DOCUMENTATION_BOUNDARIES.md`.

## 7. Базовая архитектура стандартизации

Стандартизация framework — управляемый процесс объединения характеристик, а не полностью автоматический нормализатор.

Базовая цепочка:

* определить смысл целевой характеристики;
* выполнить read-only поиск названий характеристик;
* сформировать список кандидатов;
* вручную выбрать каноническую характеристику;
* явно решить, какие aliases включать или исключать;
* выполнить inventory исходных значений;
* утвердить каноническую единицу и контракт `normalized_value`;
* сгенерировать предложения по нормализации;
* провести отдельную ручную проверку;
* подготовить отдельный явный apply-plan;
* применять изменения только после соответствующего gate.

Framework не должен автоматически объединять похожие характеристики только по названию.

Статус `approved` в review-цепочке не означает разрешение выполнять SQL/apply.

## 8. Gates и безопасность

По умолчанию без отдельного явного `+` запрещены:

* file changes и implementation;
* подключение к БД;
* pipeline run;
* SQL preview;
* apply-plan;
* SQL/apply;
* product/category data changes;
* production actions;
* cache rebuild;
* commit и push.

Разрешение на один bounded step не переносится автоматически на следующий шаг и не расширяет scope.

Protected user-owned changes нельзя изменять, stage, commit, restore, clean или использовать как основание для reset рабочего дерева.

Смысл характеристик и единиц измерения нельзя угадывать автоматически.

Характеристики, связанные с selector/cache, требуют явного контракта канонической единицы до реализации.

Пересборка cache требует отдельного явного подтверждения.

## 9. Рабочие технические правила

PowerShell-команды для пользователя давать одной строкой, совместимой с PowerShell 7+.

PHP-проверки выполнять через:

`C:\php56\php.exe`

Codex обычно не делает commit в одном шаге с реализацией, если это явно не разрешено prompt. После отчёта Codex ChatGPT сначала проверяет scope и diff.

## 10. Чем START_HERE.md не является

`START_HERE.md`:

* не является prompt для Codex;
* не является status-документом;
* не является changelog или backlog;
* не заменяет `HANDOFF.md`;
* не заменяет `DECISIONS.md`, `RUNTIME_CHECKS.md` или specs;
* не содержит текущую рабочую задачу;
* не даёт authorization на изменения.

Его задача — обеспечить одинаковый и безопасный вход новой ChatGPT-сессии и правильную передачу bounded work в Codex.
