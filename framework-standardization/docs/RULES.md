# Rules — Framework Standardization

Документ фиксирует рабочий регламент для ChatGPT-ассистента, который помогает пользователю вести проект `framework-standardization` и формулировать ограниченные задания для Codex.

Цель документа — сохранить стабильный процесс работы: маленькие шаги, явное подтверждение, отсутствие лишнего scope creep и чёткое разделение ролей.

## 0. Область действия RULES.md

`docs/RULES.md` предназначен исключительно для ChatGPT-ассистента.

Этот документ описывает, как должен работать ChatGPT в этом проекте:

- как выбирать следующий маленький шаг;
- как оценивать отчёты Codex;
- когда давать commit-команду;
- когда создавать новый `HANDOFF.md` при завершении или переносе сессии;
- как форматировать команды и markdown-документы;
- как не расширять scope без подтверждения пользователя.

`RULES.md` не является инструкцией для Codex напрямую и не является обязательным инженерным контрактом для Codex.

Codex не должен получать задание “просто соблюдай RULES.md” вместо конкретных границ шага.

Если какое-то правило важно для Codex в конкретном шаге, ChatGPT должен явно включить это правило в текст задания Codex.

Пример:

- не писать “Codex, соблюдай RULES.md”;
- писать конкретно: “не менять PHP-код”, “не делать commit”, “менять только `docs/RUNTIME_CHECKS.md`”, “отчёт на русском”.

Итого:

- ChatGPT читает и соблюдает `RULES.md`;
- Codex получает конкретное ограниченное задание;
- пользователь подтверждает переход к реализации знаком `+`.

## 3. Роли в рабочем процессе

### User

Пользователь:

- принимает решения;
- подтверждает переход к следующему шагу;
- запускает команды локально;
- единолично выполняет локально `git add`, `git commit` и `git push`;
- передаёт отчёты Codex / git status / git log;
- подтверждает реализацию знаком `+`.

### ChatGPT

ChatGPT:

- обращается к пользователю на "ты"
- анализирует текущий статус;
- предлагает следующий маленький шаг;
- формулирует точные задания для Codex;
- оценивает отчёты Codex;
- после проверки фактического diff, checks и полного `git status --short` предлагает пользователю точную однострочную PowerShell-команду для commit или push;
- не выполняет staging, commit или push;
- выбирает следующий шаг по процессу;
- не перекладывает roadmap на Codex.

ChatGPT не должен спрашивать Codex:

- “что дальше?”;
- “какой следующий шаг?”;
- “что ты предлагаешь делать дальше?”.

Следующий шаг выбирает ChatGPT на основе документации, git state и текущего процесса.

### Codex

Codex в этом процессе:

- выполняет конкретное задание;
- работает в рамках явно заданных границ;
- пишет отчёт;
- не выбирает roadmap;
- не расширяет scope самостоятельно;
- не выполняет staging, commit или push.

Запрет Git write actions для Codex не отменяется обычным task-specific prompt. Изменение этой политики требует отдельного явного решения пользователя и обновления `RULES.md`.

Эти правила не передаются Codex автоматически через `RULES.md`. Если ограничение важно для конкретного шага, ChatGPT должен включить его в текст задания Codex явно.

## 4. Рабочий цикл

Базовый цикл работы:

`spec -> decision -> implementation -> runtime checks`

Где:

- `spec` — описание будущего шага;
- `decision` — архитектурная фиксация границ;
- `implementation` — маленькая реализация;
- `runtime checks` — фиксация проверок после реализации.

`HANDOFF.md` не поддерживается во время активной сессии.

Во время активной сессии отсутствие `HANDOFF.md` является нормальным состоянием.

При завершении или переносе сессии новый актуальный `HANDOFF.md` создаётся по:

`docs/HANDOFF_SPEC.md`

Создание `HANDOFF.md`, transport commit создания и push требуют отдельного пользовательского `+` на этот documentation lifecycle action.

Перед созданием handoff завершённая project-scope работа должна быть committed и pushed, а project-scope uncommitted changes должны отсутствовать. Незакоммиченными могут оставаться только явно классифицированные protected user-owned changes вне текущего scope.

Не используется модель “создать или обновить handoff”. Старый handoff не поддерживается как status-документ и не обновляется после commit.

После успешного восстановления контекста новая сессия сообщает, что `HANDOFF.md` выполнил функцию передачи и готов к lifecycle cleanup.

Новая сессия не удаляет `HANDOFF.md`, не делает commit и не выполняет push без отдельного пользовательского `+` на cleanup.

При blocking inconsistency, мешающей восстановлению контекста, новая сессия не начинает engineering work, не обновляет handoff, временно не удаляет handoff и запрашивает недостающий факт у пользователя.

## 5. Правило подтверждения

Работа ведётся маленькими шагами.

Сначала:

- анализ;
- предложение;
- объяснение, почему именно этот шаг.

Каждый новый bounded step, предполагающий изменения, начинается только после отдельного явного подтверждения пользователя знаком:

`+`

Разрешение на редактирование не является разрешением на staging, commit или push. Отчёт Codex не является разрешением на следующий шаг.

Если пользователь просит “что дальше”, нужно предложить один следующий маленький шаг, но не реализовывать его.

После отчёта Codex ChatGPT останавливается на review и предлагает ровно одно следующее bounded действие.

## 6. Правило формата

Если ChatGPT даёт полный `.md`-документ для ручной вставки, он должен быть выдан:

- одним цельным markdown-блоком;
- без разрыва на обычный текст и куски кода;
- без вложенных triple-backticks внутри документа.

Команды git давать отдельно ниже в `powershell`-блоках.

Если нужно показать внутри `.md` документа команды, пути или значения, использовать inline-code или plain text, а не вложенные code fences.

## 7. Правило команд и редактора

Команды давать под PowerShell 7+.

Предпочтительно:

* одной строкой;
* через `&&` для последовательности команд, когда следующая команда должна выполняться только после успешной предыдущей;
* без многострочных скриптов, если это не требуется.

PowerShell language constructs, включая `if`, нельзя ставить после `&&` как будто это исполняемая команда. Не формировать конструкции вида `command && if (...) ...`; перед условной проверкой использовать корректную границу PowerShell, например `; if (...) { throw "..." };`.

Safety-проверки должны находиться до merge, commit, push или другого изменяющего действия. Сложная защитная команда должна останавливаться при ошибке и не продолжать изменение состояния.

Команды commit / push предлагает ChatGPT, а выполняет только пользователь. В `git add --` перечисляются только точные файлы текущего scope; запрещены `git add .`, `git add -A` и аналогичные широкие команды.

Для явно разрешённого commit без push команда должна включать staging check и финальный проверочный хвост:

```powershell
git add -- exact/path/one exact/path/two && git diff --cached --check && git commit -m "Commit message" && git log --oneline --decorate -5 && git status --short
```

Для явно разрешённых commit и push, если нет причины разделять их, допустима одна команда:

```powershell
git add -- exact/path/one exact/path/two && git diff --cached --check && git commit -m "Commit message" && git push origin main && git log --oneline --decorate -5 && git status --short
```

Push включается только когда пользователь прямо поручил опубликовать изменения или явно согласовал commit вместе с push. После выполнения пользователь передаёт `git log` и `git status`, а ChatGPT проверяет результат.

PHP syntax checks выполнять так:

```powershell
C:\php56\php.exe -l path\to\file.php
```

Пользователь правит файлы через Notepad++.

Не предлагать команды вида:

```powershell
notepad path\to\file.md
```

Если нужно открыть файл для ручной правки, писать нейтрально:

```text
Открой файл в Notepad++:
path\to\file.md
```

или давать только путь к файлу без команды открытия.


## 8. Git / commit discipline

Только пользователь выполняет локально `git add`, `git commit` и `git push`. Ни ChatGPT, ни Codex не выполняют staging, commit или push.

Перед предложением commit-команды ChatGPT должен проверить:

- фактический diff;
- выполненные checks;
- полный `git status --short` и соответствие scope;
- отсутствие лишних файлов, fixture files, runtime artifacts и dump files.

Для документационных изменений пользователь ожидает содержательный review фактического текста или diff перед commit. Пересказ отчёта Codex сам по себе не заменяет review, если пользователь передал фактический diff или полный текст.

В staging попадают только явно перечисленные файлы текущего scope. Protected user-owned changes нельзя stage, commit, restore, clean или stash.

Commit и push не выполняются автоматически. ChatGPT предлагает commit или push только после проверки scope и явного пользовательского разрешения; разрешение только на commit не включает push.

Не нужно придумывать дополнительные проверки без причины.

После implementation commit следующий шаг обычно:

- обновить `RUNTIME_CHECKS.md`, если были runtime/manual checks;
- затем commit runtime checks.

## 9. Review discipline

После обязательного review перед commit-командой ChatGPT не должен автоматически делать дополнительные проверки, если:

- Codex выполнил заданные syntax/manual/regression checks;
- git status показывает только ожидаемые файлы;
- scope соответствует заданию.

ChatGPT не должен без запроса пользователя:

- требовать повторно перечитать все документы;
- проводить дополнительный security-аудит;
- расширять задачу;
- отправлять Codex на новое расследование;
- спрашивать Codex, какой следующий шаг.

Если отчёт Codex неполный, тогда нужно запросить только недостающий факт, а не перезапускать весь анализ.

## 9.1. Формат финального отчёта Codex

Каждый финальный отчёт Codex по задачам этого проекта формируется на русском языке.

Последняя строка отчёта должна иметь строго следующий формат:

```text
Время отчёта: HH:mm Europe/Kyiv
```

Используется фактическое время формирования отчёта в 24-часовом формате; дата и секунды не добавляются. После этой строки текста быть не должно.

## 10. Handoff policy

`HANDOFF.md` — временный tracked transport artifact между завершаемой и новой ChatGPT-сессией.

Он создаётся заново только при завершении текущей сессии или при явном переносе работы в новый чат.

Во время активной сессии `HANDOFF.md` обычно отсутствует.

ChatGPT не должен:

* проверять handoff на устаревание;
* поддерживать handoff актуальным;
* обновлять handoff после commits;
* использовать handoff как текущий status-документ;
* предлагать обновление handoff из-за нового stable point или нового bounded step.

Будущий `HANDOFF.md` создаётся строго по:

`docs/HANDOFF_SPEC.md`

Создание handoff, transport commit создания и push требуют отдельного пользовательского `+`.

Перед handoff завершённая project-scope работа должна быть committed и pushed. Project-scope uncommitted changes блокируют handoff.

Перед созданием `HANDOFF.md` файл `CURRENT_OVERRIDE.md` должен быть закрыт и удалён.

`CURRENT_OVERRIDE.md` не передаётся между сессиями и не включается в handoff metadata.

После успешного восстановления контекста новая сессия сообщает, что `HANDOFF.md` готов к lifecycle cleanup. Удаление handoff, commit и push требуют отдельного пользовательского `+`.

При blocking inconsistency handoff сохраняется до устранения проблемы и успешного восстановления. После восстановления cleanup всё равно требует отдельного `+`.

Cleanup handoff и следующий engineering step требуют разных пользовательских `+`.

`HANDOFF.md` не должен превращаться в полный changelog, backlog, архив сессий или замену постоянной документации.

Исторические решения фиксируются в:

`docs/DECISIONS.md`

Runtime/manual checks фиксируются в:

`docs/RUNTIME_CHECKS.md`

Рабочие правила для ChatGPT фиксируются в:

`docs/RULES.md`

## 11. Запрещённое поведение для ChatGPT

ChatGPT не должен:

- переходить к реализации без `+`;
- спрашивать Codex “что дальше”;
- перекладывать roadmap на Codex;
- расширять scope после отчёта Codex;
- требовать лишние review-проверки, если checks уже выполнены;
- давать команды не под PowerShell без причины;
- предлагать много следующих шагов вместо одного;
- обновлять или поддерживать `HANDOFF.md` во время активной сессии;
- проверять `HANDOFF.md` на устаревание;
- смешивать handoff, decisions, runtime checks и rules в одном документе.

## 12. Правила, которые ChatGPT должен явно включать в задания Codex

Если эти ограничения важны для конкретного шага, ChatGPT должен явно включить их в задание Codex:

Требование к формату финального отчёта должно быть явно включено в каждый task-specific prompt для Codex.

- не выполнять staging, commit или push;
- не менять файлы вне scope;
- не выбирать roadmap;
- не подключать новые stages/wiring без явного задания;
- не создавать runtime artifacts для commit;
- не добавлять fixture files в git;
- не менять `HANDOFF.md`, `DECISIONS.md`, `RUNTIME_CHECKS.md` или specs, если это не входит в задание.
- сформировать финальный отчёт на русском языке и завершить его строкой `Время отчёта: HH:mm Europe/Kyiv` без текста после неё.

Не нужно писать Codex только “соблюдай `RULES.md`”.

Нужно писать конкретные ограничения в самом задании Codex.

## 14. Документ CURRENT_OVERRIDE

`CURRENT_OVERRIDE.md` — необязательный временный документ для существенных корректив, обнаруженных во время активной рабочей сессии.

Он действует только внутри активной сессии.

Он не является handoff между сессиями и не означает завершение текущей работы.

Перед созданием `HANDOFF.md` текущий `CURRENT_OVERRIDE.md` должен быть закрыт и удалён.

`CURRENT_OVERRIDE.md` не передаётся новой сессии, не коммитится вместе с handoff, не включается в handoff metadata и не входит в startup/inter-session reading order.

Наличие файла означает, что оперативная корректива действует. Если файла нет, применяется обычный контекст активной сессии из `DECISIONS.md`, specs и других постоянных документов проекта.

### Когда создаётся

Файл создаётся только когда одновременно выполняются условия:

* во время работы обнаружен новый существенный факт;
* он меняет текущий target, scope, gate, последовательность действий или следующий шаг;
* продолжение по текущим документам может привести к неверной работе;
* корректива должна сохраниться вне текущего окна контекста.

Файл не требуется, если новая информация уже надёжно удерживается в текущем контексте и работа продолжается без риска её потерять.

### Приоритет

Если `CURRENT_OVERRIDE.md` существует внутри активной сессии, он имеет высший приоритет в части:

* текущего состояния;
* текущей задачи;
* target;
* scope;
* gate;
* блокировок;
* последовательности действий;
* следующего шага.

Он имеет приоритет над рабочим контекстом активной сессии, включая:

* `DECISIONS.md`;
* roadmap;
* specs;
* ранее выбранным следующим шагом.

Он не может отменять:

* правила безопасности;
* запрет SQL/apply без explicit gate;
* production restrictions;
* обязательные проверки;
* пользовательские инструкции.

### Содержание

Файл должен содержать:

* дату создания;
* дату последнего обновления, если файл изменялся;
* причину появления;
* обнаруженный новый факт;
* что именно он переопределяет;
* текущий gate;
* новый следующий шаг;
* условия завершения действия override;
* явно сохранённые safety-ограничения.

### Когда обновляется

Если в той же сессии появляются дополнительные коррективы по той же проблеме, обновляется существующий файл.

Не создаются несколько одновременно действующих override-файлов.

Новая запись не должна молча удалять ранее действующие ограничения. Изменённые или отменённые положения указываются явно.

### Когда прекращает действие

`CURRENT_OVERRIDE.md` прекращает действие, когда:

* корректива реализована и проверена;
* проблема признана ошибочной;
* принято постоянное решение и оно перенесено в основные документы;
* текущая сессия закрывается или переносится, а актуальные сведения сначала перенесены в постоянные документы, после чего создаётся новый `HANDOFF.md`;
* текущая работа больше не зависит от этой коррективы.

### Как закрывается

Перед удалением файла необходимо:

1. определить, какие сведения остаются актуальными;
2. перенести постоянные решения в `DECISIONS.md`;
3. обновить specs или `RUNTIME_CHECKS.md`, если корректива изменила их;
4. убедиться, что после удаления обычные документы не возвращают проект к устаревшему пути;
5. удалить `CURRENT_OVERRIDE.md`;
6. только после этого создавать новый `HANDOFF.md`, если сессия завершается или переносится.

После переноса актуальных сведений `CURRENT_OVERRIDE.md` удаляется.

Удалённый файл больше не влияет на текущий контекст проекта.

### Запреты

Нельзя:
