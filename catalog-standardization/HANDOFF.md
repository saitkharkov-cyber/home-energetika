# HANDOFF.md

## Контекст

В ходе работы произошло важное архитектурное переосмысление проекта.

Изначально вся разработка велась внутри проекта **Pump Selector**. По мере развития стало очевидно, что задачи стандартизации каталога образуют самостоятельную предметную область со своей архитектурой, документами, инструментами и жизненным циклом.

Поэтому принято решение выделить **Catalog Standardization** в самостоятельную подсистему внутри общего проекта **Home-Energetika**.

---

# Текущее состояние

Стандартизация ведется для категории:

**Скважинные насосы**

Работа выполняется строго по одному атрибуту за раз.

Для каждого атрибута используется единый pipeline:

1. Анализ.
2. Выявление шаблонов.
3. Проверочный SELECT.
4. UPDATE.
5. Контрольный SELECT.
6. Проверка исключений.
7. Обновление названия атрибута.
8. Обновление Catalog_Standardization.xlsx.

Переход к следующему атрибуту допускается только после полного завершения предыдущего.

---

# Что завершено

Для категории "Скважинные насосы" полностью стандартизированы:

* Максимальная производительность (л/мин)
* Напряжение (В)
* Максимальный напор (м)

Для этих атрибутов выполнены:

* стандартизация значений;
* проверка исключений;
* контрольные SELECT;
* приведение названий к принятому стандарту.

# Важное архитектурное открытие

В конце сессии выявлена архитектурная ошибка.

Работы по стандартизации каталога выполнялись внутри проекта Pump Selector.

После анализа принято решение разделить предметные области.

Целевая структура:

```text
home-energetika/
│
├── pump-selector/
│
└── catalog-standardization/
```

Pump Selector и Catalog Standardization являются самостоятельными подсистемами.

Подборщик использует результаты стандартизации, но не отвечает за неё.

---

# ПЕРЕД ПРОДОЛЖЕНИЕМ РАБОТ (ОБЯЗАТЕЛЬНО)

Не продолжать стандартизацию каталога до наведения порядка в репозитории.

В первую очередь необходимо:

1. Спроектировать новую структуру проекта.
2. Разделить материалы Pump Selector и Catalog Standardization.
3. Переместить документацию, скрипты и рабочие файлы в соответствующие подсистемы.
4. Создать отдельные ветки разработки:

   * Pump Selector;
   * Catalog Standardization.
5. Настроить структуру Git и `.gitignore`.
6. Только после этого выполнять новые коммиты.

Текущие изменения **не коммитить**, чтобы не закрепить ошибочную структуру репозитория.

---

# Точка продолжения

После завершения реорганизации репозитория:

1. Подготовить DATA_PIPELINE.md.
2. Обновить структуру каталогов.
3. Выполнить первые архитектурные коммиты.
4. Вернуться к стандартизации категории "Скважинные насосы".

Следующий атрибут:

**Диаметр насоса (мм)**

Работу продолжить по стандартному pipeline, принятому в CATALOG_STANDARD.md.
