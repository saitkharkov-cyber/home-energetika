# Стандарт каталога

## Назначение

Настоящий документ определяет единые правила хранения и отображения характеристик товаров.

Все новые импорты и ручные изменения должны соответствовать данному стандарту.

---

# Общие правила

## 1. Один атрибут — одна характеристика

Каждый атрибут описывает только одну физическую характеристику.

Запрещается хранить в одном значении несколько характеристик одновременно.

---

## 2. Единицы измерения

Единицы измерения указываются в названии атрибута.

Примеры:

* Максимальный напор (м)
* Максимальная производительность (л/мин)
* Диаметр насоса (мм)
* Мощность двигателя (кВт)

---

## 3. Значения

Значение должно содержать только само значение характеристики.

Допустимо:

* 115
* 45
* 76
* 220

Недопустимо:

* 115 м
* 45 л/мин
* 220 V
* 230 В / 50 Гц (однофазный)

---

## 4. Канонический атрибут

Для каждой характеристики существует только один допустимый attribute_id.

Создание дублирующих атрибутов запрещается.

======

# Принципы стандартизации

При возникновении неоднозначных ситуаций принимаются решения в соответствии со следующими принципами.

## 1. Простота важнее сложности

Если поставленная задача решается без усложнения архитектуры, выбирается более простое решение.

---

## 2. Проверка важнее предположений

Любая миграция выполняется только после анализа существующих данных.

Предположения не являются основанием для изменения каталога.

---

## 3. Один источник истины

Для каждой характеристики существует один:

* Attribute ID;
* формат названия;
* формат значения.

---

## 4. Атомарность данных

Каждый атрибут хранит только одну характеристику.

Каждое значение хранит только одно значение характеристики.

---

## 5. Стандартизация важнее совместимости с ошибками

Если исторические данные противоречат принятому стандарту, данные должны быть приведены к стандарту после проверки их корректности.

---

## 6. Документ является нормативным

При возникновении новых импортов, ручного заполнения или разработки модулей данный стандарт имеет приоритет над существующей практикой заполнения каталога.


======

# Уровни стандартизации

Стандартизация каталога выполняется независимо по четырем уровням.

## Уровень 1. Названия атрибутов

Для каждой характеристики существует одно каноническое название.

Пример:

* Напряжение (В)
* Максимальный напор (м)
* Диаметр насоса (мм)

---

## Уровень 2. Значения атрибутов

Каждое значение хранится в едином формате.

Примеры:

* 220
* 380
* 115
* 76

---

## Уровень 3. Attribute ID

Для каждой характеристики существует один канонический `attribute_id`.

Использование нескольких `attribute_id` для одной характеристики не допускается.

---

## Уровень 4. Семантика

Каждый атрибут должен описывать только одну физическую характеристику.

Запрещается объединять в одном атрибуте несколько различных характеристик или технических описаний.

======

# Признаки нарушения стандарта

Каталог считается нарушающим стандарт, если выполняется хотя бы одно из условий.

## 1. Дублирование Attribute ID

Одна и та же характеристика хранится в нескольких `attribute_id`.

Пример:

* Напряжение → ID 15 и ID 57.

---

## 2. Дублирование названий

Одинаковые названия принадлежат разным `attribute_id`.

Пример:

* Тип насоса → 6 различных `attribute_id`.

---

## 3. Различные названия одной характеристики

Одна физическая характеристика имеет несколько названий.

Пример:

* Напряжение
* Напряжение сети
* Напряжение питания

---

## 4. Нестандартизированные значения

Значения одной характеристики записаны в разных форматах.

Пример:

* 220
* 220 В
* 220V

---

## 5. Неатомарные значения

Одно значение содержит несколько характеристик или описание вместо одного значения.

Недопустимые примеры:

* 230 В / 50 Гц (однофазная)
* 400 В / 50 Гц (трёхфазный)
* 220–240 В

Каждое значение должно описывать только одну характеристику.

======

# Порядок стандартизации

Работы по стандартизации каталога всегда выполняются в следующем порядке.

## Шаг 1. Инвентаризация

Определить:

* используемые Attribute ID;
* названия атрибутов;
* количество товаров;
* используемые значения.

На этом этапе изменения в каталог не вносятся.

---

## Шаг 2. Анализ

Для каждого атрибута определить:

* существуют ли дубли;
* существуют ли разные названия одной характеристики;
* существуют ли различные форматы значений;
* существуют ли нарушения семантики.

---

## Шаг 3. Утверждение стандарта

До внесения изменений определить:

* канонический Attribute ID;
* каноническое название;
* канонический формат значения.

Только после утверждения допускается миграция данных.

---

## Шаг 4. Миграция

Внести изменения в каталог.

Каждая миграция должна затрагивать только одну логическую группу характеристик.

После каждой миграции выполняется проверка результата.

---

## Шаг 5. Проверка

Проверить:

* количество товаров;
* отсутствие потери данных;
* отсутствие дублей;
* соответствие утвержденному стандарту.

Только после успешной проверки задача считается завершенной.

======

# Каноническая модель атрибута

Каждая характеристика каталога должна быть определена следующими параметрами.

## Обязательные поля

### 1. Название

Каноническое название характеристики.

Пример:

**Напряжение (В)**

---

### 2. Attribute ID

Единственный допустимый `attribute_id`.

---

### 3. Тип значения

Допустимый тип данных.

Возможные варианты:

* Целое число
* Дробное число
* Логическое значение
* Текст (ограниченный перечень)

---

### 4. Формат хранения

Правила хранения значения.

Примеры:

* только число;
* без единиц измерения;
* без дополнительного текста;
* одинарное значение.

---

### 5. Допустимые значения

Если множество значений ограничено, оно должно быть перечислено явно.

Примеры:

Напряжение (В):

* 220
* 380

Защита от сухого хода:

* Да
* Нет

---

### 6. Недопустимые значения

При необходимости фиксируются примеры ошибочного заполнения.

Например:

Напряжение (В):

* 220 В
* 220V
* 230 В / 50 Гц
* 220–240 В

---

### 7. Правила миграции

При наличии исторических данных должны быть описаны правила их приведения к каноническому виду.
