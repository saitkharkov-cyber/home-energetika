# Catalog Standard

## 1. Назначение и область действия

Этот документ задаёт нормативные правила характеристик каталога OpenCart в `catalog-standardization-new/`. Он используется parsers, matchers, generators и ручным review как источник правил данных.

Документ не описывает состояние текущей сессии, не даёт автоматического разрешения на production apply и не заменяет inventory фактической базы перед каждой операцией. Фактическая структура БД, scope товаров и наличие атрибутов должны перепроверяться перед каждым изменением.

## 2. Основные термины

### Source value

Исходное значение из сайта производителя, каталога, документа, выгрузки или OpenCart.

### Normalized value

Значение, приведённое к нормативному формату без изменения подтверждённого смысла.

### Canonical attribute

Утверждённый целевой атрибут OpenCart для конкретной характеристики.

### Alias attribute

Существующий неканонический атрибут, который может содержать ту же характеристику, но не становится canonical автоматически.

### Evidence

Проверяемое основание для значения: официальный источник, надёжный каталог, документация, подтверждённое название модели или иной явно зафиксированный источник.

### Preview

Проверяемое представление планируемых изменений, не являющееся apply.

### Apply

Фактическое изменение данных после отдельного explicit gate.

## 3. Общие нормативные правила

- Одна характеристика имеет один canonical target в рамках установленного стандарта.
- Alias-атрибуты не создаются без необходимости; существующий alias не переносится автоматически только из-за похожего названия.
- Значение сохраняет подтверждённый технический смысл; нормализация формата не превращается в догадку.
- Source value желательно сохранять в preview или промежуточной записи для аудита.
- Единицы измерения размещаются в имени canonical attribute, а значение хранится без повторения единицы.
- Числовое значение хранится числом в текстовом поле без лишних слов; десятичный разделитель — точка.
- Незначащие конечные нули не добавляются без необходимости.
- Диапазон нельзя превращать в одно число без отдельного правила; приблизительное значение нельзя делать точным.
- Значение «до N» нельзя автоматически считать равным N без принятого правила для конкретной характеристики.
- Пустое, отсутствующее и нулевое значение — разные состояния; `0` допустим только как реальное подтверждённое техническое значение.
- Данные из названия товара допустимы как evidence только после проверки структуры модели и отсутствия неоднозначности.
- Conflicting sources направляются в review, а не разрешаются молча.

## 4. Правила единиц измерения

- Canonical unit определяется характеристикой, а не произвольно выбранным источником.
- Преобразование единиц допускается только по однозначной формуле.
- Исходные единица и значение должны быть доступны в preview.
- Округление допускается только по явно установленному правилу.
- Нельзя смешивать величины разной физической природы.
- `л/мин`, `м³/ч`, `кВт`, `Вт`, `мм`, `дюймы` и `220/380 В` не взаимозаменяемы без явного преобразования.
- После преобразования хранится значение в canonical unit; формулы должны быть воспроизводимы и проверяемы.

Устойчивые формулы:

- `м³/ч × 1000 / 60 = л/мин`;
- `Вт / 1000 = кВт`;
- `дюймы × 25.4 = мм`.

Применение формулы не отменяет проверку, что исходное поле действительно описывает нужную характеристику.

## 5. Нормативный реестр подтверждённых canonical attributes

| Characteristic key | Canonical attribute | Attribute ID | Canonical value format | Unit location | Status |
|---|---|---:|---|---|---|
| `max_head` | `Максимальный напор (м)` | 12 | числовое значение в метрах | в имени атрибута | `LOCKED` |
| `flow_rate` | `Максимальная производительность (л/мин)` | 13 | числовое значение в литрах в минуту | в имени атрибута | `LOCKED` |
| `power_kw` | `Мощность двигателя (кВт)` | 14 | числовое значение в киловаттах | в имени атрибута | `LOCKED` |
| `voltage` | `Напряжение (В)` | 15 | нормализованное целевое значение напряжения | в имени атрибута | `LOCKED` |
| `diameter` | `Диаметр насоса (мм)` | 44 | числовое значение в миллиметрах | в имени атрибута | `LOCKED` |

Названия и ID основаны на подтверждённых устойчивых правилах проекта. Перед каждым изменением они перепроверяются по фактической БД, связанным нормативным документам и явно принятым более новым решениям.

## 6. Правила по подтверждённым характеристикам

### 6.1 Максимальный напор

- Canonical unit: метры.
- Хранить только числовое значение.
- Не смешивать максимальный напор с рабочим, номинальным или рекомендуемым напором.
- Значение из диапазона или графика требует отдельной интерпретации и review.
- Не выводить напор из мощности или модели без подтверждённого правила.

### 6.2 Максимальная производительность

- Canonical unit: литры в минуту.
- Хранить числовое значение.
- Допускается однозначное преобразование из `м³/ч`.
- Не смешивать максимальную, номинальную и рабочую производительность.
- Значения «до», диапазоны и точки графика требуют явно установленного правила.
- Не переносить расход из несвязанного рабочего режима.

### 6.3 Мощность двигателя

- Canonical unit: киловатты.
- Хранить числовое значение.
- Допускается преобразование из ватт делением на `1000`.
- Не смешивать потребляемую мощность, мощность двигателя, выходную мощность и мощность всей установки без подтверждения.
- Обозначения `P1`, `P2`, `input power` и `motor power` требуют review источника.

### 6.4 Напряжение

- Canonical unit: вольты; единица хранится в имени атрибута.
- Целевое значение отражает фактическое питание оборудования.
- Подтверждённые нормализованные значения `220` и `380` допустимы, когда источник однозначно описывает однофазное или трёхфазное питание.
- Нельзя автоматически заменять любое `230` на `220` или `400` на `380` без принятого правила для конкретного scope.
- Записи `220/380`, диапазоны напряжения и multi-voltage требуют review.
- Фазность нельзя придумывать только из напряжения, если источник неоднозначен.

### 6.5 Диаметр насоса

- Canonical unit: миллиметры.
- Хранить числовое значение.
- Допускается преобразование из дюймов умножением на `25.4`.
- Различать физический наружный диаметр, условный типоразмер серии и диаметр подключения.
- Обозначение `3"`, `4"` или `6"` в серии модели не всегда равно точному наружному диаметру.
- Типоразмер серии применяется только при подтверждённом соответствии конкретному физическому диаметру.
- Конфликт между фактическим диаметром и условным типоразмером требует review и фиксации evidence.

## 7. Alias attributes

- Alias-атрибут — источник кандидатов, а не автоматический target.
- Совпадение имени недостаточно.
- Перед миграцией alias в canonical проверяются физический смысл, единицы, формат, scope товаров, наличие canonical value, конфликт значений и дубликаты.
- Если canonical target уже заполнен, его нельзя перезаписывать без отдельного review.
- Перенос может быть insert-only, update-only или mixed только после preview.
- Удаление alias-атрибута не входит в обычный перенос и требует отдельного решения.
- Список исторических alias IDs не хранится здесь как универсальная истина.

## 8. Правила matching и evidence

- Exact string match не гарантирует соответствие товара.
- Нормализация названия модели должна быть воспроизводимой.
- Manufacturer, series, model, phase, power, suffix и другие значимые признаки нельзя терять.
- Один source record не назначается автоматически нескольким товарам.
- Один товар не получает несколько conflicting source records.
- Результаты делятся минимум на `matched`, `ambiguous`, `unmatched` и `rejected`.
- `ambiguous` и `rejected` не допускаются к автоматическому apply.
- Причина matching должна быть видна в preview.
- Confidence score помогает review, но не заменяет evidence.

## 9. Preview contract

Рекомендуемый минимальный набор полей preview:

- `product_id`;
- `product_name`;
- `manufacturer`;
- source identifier или URL;
- `source_model`;
- source attribute/value;
- `normalized_value`;
- `target_attribute_id`;
- `target_attribute_name`;
- `existing_target_value`;
- `proposed_value`;
- `match_status`;
- `preview_status`;
- reason или evidence note.

Состав расширяется для конкретной задачи. Preview должен позволять отличить `insert`, `update`, `skip`, `conflict` и `rejected`. Preview-файл обычно является временным рабочим результатом и не входит в Git. Preview не является разрешением на apply.

## 10. Apply contract

- Apply строится только по reviewed preview.
- Scope явно ограничен.
- SQL воспроизводим из утверждённых данных.
- Перед apply проверяются количество строк и затрагиваемые `product_id`.
- Массовый `UPDATE` без точного scope запрещён.
- Нельзя перезаписывать canonical values вне утверждённого списка.
- Insert учитывает уникальность сочетания товар–атрибут–язык.
- Язык данных задаётся явно и проверяется по фактической БД.
- Production apply выполняет пользователь после отдельного разрешения.
- Cache rebuild, reindex и иные последствия выполняются отдельно при необходимости.
- После apply обязательна verification.
- Значимое изменение должно иметь rollback или восстановимый исходный набор данных.

Конкретное имя production DB или хостинга в постоянный стандарт не встраивается.

## 11. Post-apply verification

Минимальные проверки:

- ожидаемое число затронутых товаров;
- число строк canonical attribute;
- distinct product count;
- отсутствие неожиданных duplicates;
- отсутствие пустых значений;
- отсутствие значений с повторёнными единицами;
- отсутствие неразрешённых форматов;
- сравнение affected `product_id` с утверждённым scope;
- ручная проверка нескольких representative products;
- проверка влияния на selector, если характеристика им используется.

Существенные выполненные проверки фиксируются в `docs/RUNTIME_CHECKS.md`, а не в `CATALOG_STANDARD.md`.

## 12. Добавление нового canonical attribute

1. Провести inventory существующих атрибутов.
2. Определить физический смысл.
3. Установить canonical unit.
4. Проверить фактические attribute ID и название.
5. Исследовать aliases и конфликты.
6. Определить допустимый value format.
7. Определить conversion rules.
8. Подготовить preview на реальных товарах.
9. Провести review.
10. Получить отдельное решение.
11. Добавить нормативную запись в `CATALOG_STANDARD.md`.
12. Только затем использовать атрибут как `LOCKED` canonical target.

Новый атрибут нельзя объявлять canonical только потому, что он удобен для одного парсера. Изменение `LOCKED`-записи требует отдельного решения и проверки всех зависимых процессов.

## 13. Запрещённые практики

- Придумывать значение по неполному названию.
- Молча выбирать один из conflicting sources.
- Автоматически применять ambiguous matching.
- Хранить единицу одновременно в имени и значении.
- Записывать `5 м`, если canonical format требует `5`.
- Смешивать максимальные и номинальные характеристики.
- Округлять без правила.
- Заменять заполненный canonical target без review.
- Создавать новый alias вместо использования утверждённого canonical attribute.
- Выполнять production apply по сырому parser output.
- Считать preview доказательством корректности.
- Заносить временные результаты и DB exports в Git.
- Использовать этот документ как журнал текущей работы.
