# Project Master Summary

## 1. Проект

- Репозиторий: home-energetika.
- Активная область: catalog-standardization-new/.
- Назначение: получение, проверка, сопоставление и безопасный импорт характеристик производителей в каталог OpenCart.

Результат работы — проверенные данные и воспроизводимые специализированные инструменты для конкретных производителей и задач. Эта область не является универсальной платформой или готовым framework.

## 2. Проблема, которую решает проект

Характеристики производителей поступают из разных источников, а названия моделей и форматы значений различаются. Данные сайта могут быть неполными, устаревшими или неоднозначными. Автоматическое совпадение по названию небезопасно, а ошибочный импорт может повредить каталог и работу selector.

Поэтому процесс разделяет получение исходных данных, нормализацию, сопоставление товаров, проверку, генерацию preview и разрешённое применение изменений.

## 3. Область ответственности

В область входят:

- изучение источников производителя;
- специализированный парсинг;
- подготовка нормализованных записей;
- сопоставление записей источника с товарами OpenCart;
- выявление ambiguous и unmatched случаев;
- preview и отчёты;
- генерация SQL или другого apply-артефакта после отдельного gate;
- фиксация существенных проверок.

В область не входят:

- исправление логики Pump Selector;
- ручная стандартизация всех категорий;
- разработка универсальной ETL-платформы;
- автоматический production apply без review;
- хранение секретов, дампов БД и временных файлов в Git.

## 4. Границы с соседними областями

### catalog-standardization/

Историческая область. Может служить источником старых скриптов, наблюдений и lessons learned, но не является активной областью новых изменений без отдельного разрешения.

### docs/standardization/manual-toolkit/

Самостоятельная ветка ручной стандартизации категорий, работающая по собственному процессу. Она не входит в catalog-standardization-new/.

### pump-selector/

Потребитель стандартизированных характеристик. Он не отвечает за исправление или импорт исходных данных.

### framework-standardization/

Экспериментальная область с более широкой архитектурой. Отдельные правила рабочей дисциплины могли быть адаптированы, но сама framework-архитектура автоматически не переносится.

## 5. Основной workflow

1. Определить bounded task и scope.
2. Провести inventory источников, товаров и текущих атрибутов.
3. Получить исходные данные.
4. Нормализовать значения без потери исходного материала.
5. Выполнить matching.
6. Разделить результаты на matched, ambiguous, unmatched и rejected.
7. Подготовить preview.
8. Провести ручной review.
9. Получить explicit gate.
10. Сгенерировать или выполнить разрешённое изменение.
11. Провести post-apply verification.
12. Зафиксировать существенный runtime check.

Это концептуальная последовательность, а не утверждение о полной автоматизации всех этапов.

## 6. Типы инструментов

### Parsers

Получают данные из конкретного источника или формата.

### Matchers

Сопоставляют записи источника с товарами каталога и обязаны сохранять неоднозначность, а не скрывать её.

### Generators

Создают preview, SQL или иной apply-артефакт на основании уже проверенных данных.

Специализированные инструменты допустимы и предпочтительны. Generic-компонент возможен только после доказанного повторения одинаковой задачи. Наличие каталога scripts не означает наличие готового универсального pipeline.

## 7. Модель данных и атрибуты

OpenCart является целевой системой каталога; характеристики привязываются к товарам и атрибутам каталога. Конкретные canonical attribute IDs и правила значений должны храниться в CATALOG_STANDARD.md, когда он будет создан. Этот документ не является реестром ID атрибутов.

Единицы измерения, допустимые значения и правила преобразования нельзя придумывать из названия товара без проверки источника.

## 8. Safety gates

- Inventory и analysis не дают разрешения на изменение данных.
- Preview не равен apply.
- Matching confidence не заменяет ручной review неоднозначных случаев.
- Production SQL/apply требует отдельного пользовательского разрешения.
- Cache rebuild и иные последствия изменения данных требуют отдельного решения.
- Commit и push являются отдельными действиями.
- Destructive и массовые операции запрещены без явно проверенного scope.
- Для значимых изменений должен быть предусмотрен rollback или воспроизводимый способ восстановления.

## 10. Git и локальные данные

В Git входят исходный код специализированных инструментов, постоянная документация и небольшие воспроизводимые конфигурации, если они необходимы и безопасны.

В Git не входят секреты, пароли, cookies, локальные proxy-настройки, DB dumps, разовые OpenCart exports, временные preview, временные CSV, логи выполнения и локальные рабочие результаты.

## 11. Критерии качественного результата

- Source provenance понятен.
- Преобразования воспроизводимы.
- Исходное значение не теряется без необходимости.
- Matching объясним.
- Ambiguous случаи не применяются автоматически.
- Preview позволяет проверить будущие изменения.
- Apply ограничен подтверждённым scope.
- Post-apply verification подтверждает фактический результат.
- Документация не смешивает постоянные правила и состояние текущей сессии.

## 12. Текущее состояние документации

- Каркас рабочей области создан.
- Постоянные workflow-документы существуют.
- Каталоги parsers, matchers и generators подготовлены.
- Специализированные рабочие скрипты ещё не должны считаться перенесёнными или утверждёнными.
- CATALOG_STANDARD.md будет создан отдельным bounded-шагом.
