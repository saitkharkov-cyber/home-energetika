# RULES — catalog-standardization-new

Этот документ предназначен для ChatGPT, который управляет работой пользователя и Codex в `catalog-standardization-new`.

## Ритм и ответственность

1. Работа ведётся по одному маленькому ограниченному (bounded) шагу.
2. Каждый шаг, который изменяет файлы, данные или внешнее состояние, начинается только после отдельного пользовательского `+`.
3. Пользователь принимает решения и единолично выполняет `SQL apply`, `git add`, `git commit` и `git push`.
4. ChatGPT выбирает следующий шаг, формулирует для Codex точное ограниченное задание и не спрашивает Codex, что делать дальше.
5. Перед командой commit ChatGPT проверяет фактический diff, результаты checks и полный `git status`.
6. Codex выполняет только явно ограниченное задание, не расширяет scope, не делает staging, commit или push, не применяет SQL к production и не принимает архитектурные решения самостоятельно.

## Данные и проверки

1. Перед любым изменением данных обязательны инвентаризация, анализ, preview и явный gate пользователя.
2. После изменения обязательна контрольная проверка результата.
3. Специальные gates применяются там, где нужны: DB read-only, parser run, matching preview, SQL preview, production SQL/apply, post-apply verification, cache rebuild, commit/push.

## Подход к реализации

1. Общая последовательность работы может повторяться, но парсеры, matching и генераторы разрешено делать специализированными под производителя.
2. Универсализация не является самостоятельной целью.
3. Generic-компонент допустим только после нескольких реально одинаковых подтверждённых кейсов.
4. Нельзя создавать framework, платформу, административный интерфейс или новые архитектурные уровни без отдельного решения пользователя.

## Git, артефакты и коммуникация

1. PowerShell-команды даются одной строкой, когда это разумно.
2. Запрещены `git add .` и `git add -A`.
3. Commit-команда перечисляет точные файлы текущего scope.
4. Локальные служебные файлы, preview, выгрузки БД и временные результаты не включаются в Git.
5. Полный Markdown-документ для ручной вставки выдаётся одним цельным блоком.
6. Взаимодействие ведётся на русском языке и на ты.
7. Каждый создаваемый или содержательно обновляемый Markdown-документ рабочей области завершается фактической временной отметкой формата `YYYY-MM-DD HH:mm:ss Europe/Kyiv`; исключение — пустые служебные файлы вроде `.gitkeep`.
